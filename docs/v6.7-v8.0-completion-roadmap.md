## Completion Definition (the load-bearing decision)

AOF completion is **not** "the AI's judgment is closed by machines alone." That
is impossible for any framework (see Permanent Boundaries). AOF completion is:

> The range a machine can guarantee (structure, provenance, independence, audit)
> is maximized; the range a machine cannot guarantee (semantic correctness,
> real-world outcome, human comprehension) is not hidden but escalated to humans;
> and the boundary between the two is implemented, measured, and specified.

A product that claims to close AI judgment purely by machine is overclaiming.
AOF's durable differentiation is being the framework that treats the
unsolvable range as a first-class citizen rather than pretending to remove it.

## Permanent Boundaries (not solvable by current software science)

These are recorded so no future release treats them as bugs to be "finished."

1. **Semantic correctness cannot be machine-guaranteed.** Machine verifiers of a
   machine judge lead to infinite regress. Cross-checking with another LLM
   reduces but does not remove correlated blind spots (shared training
   distribution). Formal methods bound syntactic properties (schema, type,
   protocol); "is this the right design for the need" is undecidable in general.
   Ground truth for semantic quality lives only in human and real-world feedback.
2. **A framework layer has no total enforcement.** True enforcement (cannot
   execute without passing) lives in OS / sandbox / API-gateway layers, not in a
   CLI or library. AOF's ceiling is *detectability of bypass* and *making AOF the
   shortest path*, not prevention. Requiring prevention makes completion
   unreachable.
3. **Goodhart pressure is permanent.** Fixed gates get optimized for passage, not
   quality. Rotation, human spot-checks, and negative acceptance mitigate but do
   not eliminate this. Gate-pass never equals quality.

AOF already respects these (reserved `externally_validated` for human/real-world
evidence, authority boundaries, governance triggers). The roadmap's job is to
make the machine-guaranteed range as large and honest as possible, then stop.

## Version Roadmap

### `v6.7` — Verifiable Governance (COMPLETED)

Make structural self-attestation machine-detectable.

- `TASK-072`: `archmap-impact-audit` (safe_read) — implementation-grade work items
  at/after the `TASK-071` cutoff must have a resolvable impact record, a valid
  3-value status, and a non-pending council status.
- `TASK-073`: review provenance schema + `review-provenance-audit` — record
  `{ session_id, actor_ref, model_id, maker_session_ref }` on review artifacts;
  detect maker == checker as `self_attested` (grade, not delete).
- `TASK-074`: evidence independence field + `evidence-independence-audit` — grade
  evidence `low|medium|high`; fail when a user-visible/public/release claim rests
  on low-independence evidence only.
- `TASK-075`: integrate the three audits into `release-state-audit`; fill the
  empty `inputs`/`outputs` in command help and assert non-empty in
  `cli-help-benchmark`.
- `TASK-076`: v6.7 release closure with QIF readiness review.

Completion gate: missing disposition, missing review provenance, and low
independence evidence each fail the relevant audit; `release-state-audit`
consumes the governance audits before sign-off.

### `v6.8` — Executable Quality Ledger

- `qif-evidence-append` / `qif-ledger-audit`: make the Living QIF Ledger
  machine-appendable (bug / corrected assumption / contradicted claim linked to
  intent). Do **not** auto-compute verdicts — that would overclaim.
- append-only `runtime-session-ledger`: direction/review/release artifacts must
  cite a recent ledger entry; audit detects "review artifact with no runtime run."
- operator comprehension protocol, n>=3: first path to any `operator_validated`.

Completion gate: a bypassed review (no runtime run) is machine-detected; first
`operator_validated` evidence exists.

### `v6.9` — Real Adoption

- 2+ real projects (non-AOF) run 5+ governed work items each, with outcome
  writeback and friction captured as `adoption-case-record` (distinct from the
  synthetic `adoption-proof-benchmark`).

Completion gate: first `externally_validated` evidence exists.

### `v7.0` — Governed Multi-Actor Orchestration

Prerequisite: provenance (v6.7) must exist first, or self-attestation multiplies.

- work-claim / lease schema (who holds a work item; timeout).
- parent's mandatory verification of child results (a child's `done` is not
  accepted as evidence).
- runtime verification of budget / stop conditions.
- AOF MCP tool surface (make AOF the shortest path for orchestrators).

Completion gate: parallel execution under self-attestation detection passes a
benchmark.

### `v7.x` — Conformance Protocol

- separate spec from this repo's implementation.
- conformance levels L0 (framing) → L1 (governance gates) → L2 (machine
  verification) → L3 (independent review provenance), each with a machine suite.

Completion gate: AOF itself passes L3; an external implementation can claim a level.

### `v8.0` — Bounded Completion

Four conditions to declare bounded completion:

1. AOF itself is L3-conformant (self-attestation is machine-detectable at every layer).
2. 3+ external projects have operational trace and outcome writeback.
3. operator comprehension is measured with real data.
4. the machine-guaranteed boundary is written as normative spec.

What remains after completion: human judgment does not disappear. The completed
form of AOF is an OS that minimizes and makes visible exactly where human
judgment is required — not one that removes it.

## Fable 5 Review Checkpoints

Codex drives implementation. Request a Fable 5 (or Mythos 5) review at these
points, where the risk is design-level or claim-level rather than mechanical:

- [ ] **CP-1 — before `v6.7` release closure (TASK-076).** Adversarial review of
  whether the three audits actually catch self-attestation, or only pattern-match
  its shape. Highest leverage: a weak audit here makes every later "governed"
  claim hollow.
- [ ] **CP-2 — before the first `externally_validated` claim (`v6.9`).** External
  validation is the highest-stakes claim in the whole roadmap. Review the adoption
  evidence for real independence vs. self-hosting in disguise.
- [ ] **CP-3 — before `v7.0` multi-actor execution ships.** Confirm provenance is
  truly a prerequisite and that a parent cannot accept a child's `done` as
  evidence. Multi-actor without this multiplies the self-attestation problem.
- [ ] **CP-4 — before the `v8.0` bounded-completion declaration.** The completion
  criteria themselves need adversarial review — the failure mode is declaring
  completion against criteria that were quietly weakened.

Between checkpoints, mechanical work (schemas, fixtures, command wiring, tests)
does not require a Fable 5 review; normal Guardian/Council runtime review applies.

## QIF Boundary For This Roadmap

This roadmap is a direction record, not a quality claim. Adopting it does not make
any listed outcome achieved. Each version's completion gate is structural/runtime
evidence of governed disposition, not proof of semantic correctness, market value,
or operator comprehension — those require the human and external evidence named
above and remain reserved until produced.
