# AOF v6.0 Release Definition

## Release

- version: `6.0.0`
- tag: `v6.0.0`
- theme: Public Runtime Readiness
- parent task: `TASK-056`

## Release Claim

`v6.0.0` makes AOF publishable as a historyless public baseline that a first-time human or AI operator can inspect, verify, and run without mistaking activity volume for quality.

The release claim is intentionally narrow:

> AOF is public-runtime-ready when its command surface, examples, viewer state, onboarding path, roadmap, release artifacts, and validation evidence are explicit and reproducible.

## Required Outcomes

- public snapshot contains no personal account references or copied Git history
- command surface is discoverable through `command-register`
- command routing alignment passes
- runtime organization, decision, release, QIF, test, and smoke checks pass
- examples explain what is seed state and what is generated by runtime
- viewer is documented as a runtime-backed visibility surface, not as a finished Human Recognition UX
- roadmap and release plan describe v6.0 and the next frontier honestly
- local tag can be produced even if GitHub push/release remains credential-blocked

## Authority Boundary

The release does not claim:

- general autonomy
- production-grade visual UX
- semantic truth of all generated artifacts
- live-provider execution for every command
- that every command has been manually exercised with every possible argument shape

The release does claim:

- the runtime command surface is registered and routable
- core verification and smoke paths are green
- public adoption risks are named rather than hidden
- v6.0 direction was passed through Need Validation before project execution

## Gates

1. Direction gate: runtime session exists and Need Validation approved the v6.0 project charter.
2. Public identity gate: personal account refs and `.DS_Store` are absent.
3. Command gate: command registry and routing audit are green.
4. Runtime gate: organization, decision, release state, QIF, test, and smoke validations pass.
5. Example gate: examples are documented as starter templates and quickstart can generate fresh artifacts.
6. Viewer gate: viewer status is explicitly bounded.
7. Release gate: package/bootstrap/active release state align with `6.0.0` and local tag `v6.0.0`.
