# AOF v9.5.0 Release Notes

## What You Can Do Now

You can require a provider incident recovery drill before a provider-backed action advances.

Before v9.5, AOF could present a controlled provider execution candidate for operator go/no-go, but the failure path was mostly a boundary statement.

After v9.5, AOF can record and audit what happens if the provider action fails: detection, containment, rollback decision, recovery action, operator notification, learning update, and whether execution remains stopped, deferred, escalated, or allowed to resume after review.

## Capability Delta

NEW:

- `PCAP-PROVIDER-INCIDENT-RECOVERY`: AOF can bind provider failure handling into a governed recovery drill.

UPDATED:

- `PCAP-PROVIDER-EXECUTION`: Provider execution candidates now have a downstream incident recovery gate before the roadmap can move toward cost/quota or production go/no-go claims.

REMOVED:

- None.

## Capability Matrix

See [docs/aof-capability-matrix.md](aof-capability-matrix.md).

In thirty seconds:

> v9.4 showed one provider-backed execution candidate. v9.5 shows what AOF will do if that candidate fails, and blocks hidden resume authority without recovery evidence.

## Value Evidence

Before:

- Operators could inspect readiness for a provider-backed action, but still had to infer the incident response path from scattered boundary text.

After:

- Operators can inspect one recovery record that says how AOF detects the incident, contains provider calls, decides rollback, recovers, notifies the operator, updates learning, and prevents resume unless governance allows it.

Reduced work:

- The operator no longer has to reconstruct failure handling from candidate, boundary, rollback, outcome, and learning artifacts manually.

Five-minute demo:

```bash
node ./src/cli.js provider-incident-recovery-audit --project .
node ./src/cli.js release-state-audit --project .
```

## Release Gate

`release-state-audit` includes `provider-incident-recovery-audit` for `9.5.0` and later.

## Not Proven

v9.5 does not prove live production provider recovery, actual rollback execution, provider semantic correctness, credential safety, billing safety, customer impact absence, third-party adoption, or market value.

It proves that incident recovery must be made reconstructable before the external runtime advances.
