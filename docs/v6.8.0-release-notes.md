# AOF v6.8.0 Release Notes

`v6.8.0` is the Executable Quality Ledger release.

It follows `v6.7.0` Verifiable Governance. v6.7 made release governance machine-checkable for Archmap impact, Council provenance, and evidence independence. v6.8 adds a narrow ledger so quality-relevant changes can be recorded and audited without pretending that records prove semantic truth.

## What Changed Since v6.7.0

- Added `quality-ledger-record`.
- Added `quality-ledger-audit`.
- Added `schemas/aof-quality-ledger-event.schema.json`.
- Added `schemas/aof-quality-ledger-audit.schema.json`.
- Added committed ledger event evidence for `TASK-078`.
- Added tests for valid ledger events and negative audit cases.
- Added command registry, AI help, and CLI reference entries for the ledger commands.
- Integrated `quality-ledger-audit` into `release-state-audit` for `6.8.0` and later releases.
- Updated the QIF provider integration docs to treat ledger events as traceability and governance evidence, not quality proof.
- Closed `TASK-078` with Council review evidence.
- Opened and closed `TASK-079` as release closure for v6.8.

## Human-Facing Meaning

v6.8 says:

> AOF should not let quality claims drift as prose. When a bug, correction, contradicted claim, confidence change, or operator feedback changes the quality picture, that change should be recordable as an auditable event.

For an operator, this means AOF can now ask:

- what quality intent or QIF package does this evidence affect?
- which task and runtime run produced or observed it?
- what evidence file supports it?
- did it change confidence, acceptance, or escalation state?
- did low-confidence or contradicted evidence reach governance review?
- is anyone accidentally claiming semantic truth from a ledger entry alone?

## Boundaries

This release does not claim:

- that ledger audit pass proves semantic correctness
- that AOF can auto-compute QIF verdicts
- that external QIF verification is replaced
- that operator/market truth is validated
- Node.js 25+ support

It claims that AOF now has a first executable ledger foundation for quality-relevant evidence changes, and that missing traceability or unsafe truth claims can be rejected before release sign-off.

## Next Frontier

After `v6.8.0`, the next frontier should be selected through runtime-backed review. Strong candidates are first-class executable pre-implementation quality gates, standalone context/reference audits, or the first bounded step toward governed multi-actor orchestration.
