# AOF v7.2 Release Definition

## Release

- version: `7.2.0`
- tag: `v7.2.0`
- theme: Work Execution Packet
- release task: `TASK-091`

## Release Claim

`v7.2.0` makes AOF execution claims harder to fake by requiring implementation-grade work to be represented as a bounded execution packet.

The bounded claim is:

> AOF can record and audit a Work Execution Packet that links a task to context integrity, actor handoff evidence, execution lineage, verification evidence, and an explicit stop/continue decision before execution or release readiness is claimed.

## Required Outcomes

- `work-execution-packet-record` writes schema-valid execution packets.
- `work-execution-packet-audit` verifies packet presence, task linkage, accepted execution status, actor handoff refs, execution lineage, verification evidence, stop/continue decision, resolvable refs, and not-proven boundary.
- `release-state-audit` includes `work-execution-packet-audit` for `7.2.0` and later releases.
- TASK-091 has work readiness, context integrity, agent session, Archmap impact, Council review, independent evidence, and Work Execution Packet evidence.
- Work Execution Packet does not replace Guardian/Council review; it packages the evidence they need to judge.

## Authority Boundary

The release does not claim:

- semantic correctness of the implemented work
- operator acceptance
- external adoption
- autonomous multi-agent execution
- live provider session streaming
- Node.js 25+ support

The release does claim:

- execution claims can be made reconstructable as a single packet
- actor handoff and verification evidence can be mandatory before execution readiness
- stop/continue decisions can be represented as first-class execution evidence
- execution packet integrity can become a release-state governance gate

## Gates

1. Contract gate: packet and audit schemas exist.
2. Writer gate: `work-execution-packet-record` creates schema-valid packet artifacts.
3. Audit gate: `work-execution-packet-audit` fails missing packets and passes TASK-091 release evidence.
4. Release integration gate: `release-state-audit` includes Work Execution Packet audit for `7.2.0` and later.
5. Governance gate: TASK-091 has work readiness, Archmap impact, Council review provenance, independent evidence, session evidence, context integrity, and packet evidence.
6. Boundary gate: Work Execution Packet is not described as semantic truth, product value, or external adoption proof.
7. Verification gate: focused runtime tests, governance audits, release-state audit, organization verification, command routing, CLI help benchmark, npm test, and smoke validation pass.
