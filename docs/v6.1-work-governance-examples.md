# AOF v6.1 Work Governance Examples

## Purpose

This document makes the `v6.1` Work Governance fixtures adoption-facing.

The examples show that AOF can govern work outside a pure software-task frame. Each example is intentionally small, but it must still answer the same operator questions:

- What work is being governed?
- Why should the work exist?
- Who or what is assigned?
- What authority does each actor have?
- What evidence is council-ready?
- What is the Go / No-Go state?
- Which operational map changed?
- Which external system remains the source of truth?
- What bounded context should the next AI or human operator read?

Mission Control may project these examples, but the source of truth is the committed runtime artifact chain under `.aof/artifacts/work-governance/fixtures/`.

## Example 1: Software Release Work

### Scenario

`SW-WORK-001` represents a governed release candidate for a customer-visible web application API change.

The work is not considered "done" because code changed or tests passed. It is governed because the release has customer impact, rollback risk, human publication authority, and downstream deployment dependencies.

### Runtime Artifact Chain

| Purpose | Artifact |
|---|---|
| Work goal | `.aof/artifacts/work-governance/fixtures/software/work-item-goal.json` |
| Actor composition | `.aof/artifacts/work-governance/fixtures/software/actor-composition.json` |
| Council-ready output | `.aof/artifacts/work-governance/fixtures/software/council-ready-output.json` |
| Go / No-Go visualization | `.aof/artifacts/work-governance/fixtures/software/go-no-go-visualization.json` |
| Operational map change log | `.aof/artifacts/work-governance/fixtures/software/operational-map-change-log.json` |
| Context pack | `.aof/artifacts/work-governance/fixtures/software/context-pack.json` |
| External source reference | `.aof/artifacts/work-governance/fixtures/software/external-ref.json` |

### Governance Reading

The objective is to prepare a governed release candidate with release evidence, an API change summary, and a rollback plan.

The actor composition selects:

- `codex` as AI builder with execute authority.
- `npm-test` as tool-backed verifier with execute authority.
- `human-owner` as publication approver with approval authority.

The authority boundary is the important point: Codex may execute and prepare release artifacts, but Codex must not decide publication approval or budget acceptance.

The council-ready output recommends `conditional_go`. This is not a release approval. It means the release evidence is mostly present, but the publication window and owner approval remain unresolved.

The operational map impact updates the decision approval flow while recording `no_change` for the system architecture map. This prevents a common false claim: a release can affect governance flow even when architecture does not change.

The external reference records GitHub as the source work item system with `comment_back_only` sync policy. AOF does not silently become the external issue tracker.

### Operator Next Action

Request human publication approval before deployment.

## Example 2: Manufacturing Quality Work

### Scenario

`MFG-WORK-001` represents a recurring packaging defect investigation.

This is deliberately non-software. The work is governed because it affects customer returns, may require inspection-flow changes, requires quality authority, and depends on an external quality management system.

### Runtime Artifact Chain

| Purpose | Artifact |
|---|---|
| Work goal | `.aof/artifacts/work-governance/fixtures/manufacturing/work-item-goal.json` |
| Actor composition | `.aof/artifacts/work-governance/fixtures/manufacturing/actor-composition.json` |
| Council-ready output | `.aof/artifacts/work-governance/fixtures/manufacturing/council-ready-output.json` |
| Go / No-Go visualization | `.aof/artifacts/work-governance/fixtures/manufacturing/go-no-go-visualization.json` |
| Operational map change log | `.aof/artifacts/work-governance/fixtures/manufacturing/operational-map-change-log.json` |
| Context pack | `.aof/artifacts/work-governance/fixtures/manufacturing/context-pack.json` |
| External source reference | `.aof/artifacts/work-governance/fixtures/manufacturing/external-ref.json` |

### Governance Reading

The objective is to investigate a recurring packaging defect and decide whether to change inspection process.

The actor composition selects:

- `plant-quality-owner` as human quality engineer with approval authority.
- `codex` as AI operations analyst with propose authority.
- `qms-export` as tool-backed evidence source with execute authority.

The composition also records a missing `compliance reviewer`. This is a first-class governance signal, not a documentation footnote. The work can proceed as an investigation packet, but process-change approval should remain blocked until compliance review exists.

The council-ready output recommends `needs_more_evidence`. This makes the example a useful counterweight to the software case: AOF must represent blocked or incomplete governed work, not only successful work.

The operational map impact updates the evidence-flow map and marks the inspection work-flow map as `needs_review`. This keeps the future process-change decision connected to the actual operating model.

The external reference records the quality management system as an `external-operational-record` with `read_only` sync policy. AOF may summarize and reason over the QMS export, but the QMS remains the source of truth.

### Operator Next Action

Assign a compliance reviewer before any inspection-process change decision.

## What These Examples Prove

These examples prove structural Work Governance coverage for `v6.1`:

- Software and non-software work can use the same governed artifact chain.
- AI, human, tool, and approval actors can appear in one work item.
- Authority boundaries prevent AI execution from becoming unauthorized approval.
- Go / No-Go state can represent `conditional_go` and `needs_more_evidence`.
- Operational map impact is explicit and reviewable.
- Context packs summarize through refs without embedding raw artifact bodies.
- External systems retain explicit source-of-truth and sync-policy semantics.

They do not prove business truth, manufacturing truth, or release-market truth. Those require human review, expert review, operational feedback, and real external system integration.
