# AOF v6.4.0 Release Notes

`v6.4.0` is the Pre-Implementation Quality Gate and QIF-Governed Explanation release.

It follows `v6.3.0` AI Command Help Surface. v6.3 made it easier for AI orchestrators to choose the right command. v6.4 addresses the next, deeper problem: AOF/QIF can still be used as records after implementation rather than as gates before implementation.

## Human-Facing Meaning

v6.4 says:

> Do not let AOF/QIF become a beautiful archive of wrong work.

Before implementation, AOF should ask:

- What user problem are we solving?
- What product assumption are we making?
- Is this a UI solution hiding a data-model problem?
- What must not happen?
- What would prove the claim from the user's visible path?
- What adjacent or downstream path could break?
- Does QIF need a new or updated Quality Intent, evidence item, confidence change, or negative acceptance?

## New Operating Rules

v6.4 formalizes these rules:

- Positive acceptance is not enough; negative acceptance is required where a forbidden state matters.
- Code reading and symmetry reasoning are low-independence evidence for user-visible/public-path claims.
- User-visible claims require user-visible E2E observation.
- Guardian must ask adjacent-path, shared-assumption, and public/irreversible-risk questions before done.
- Screenshots and DOM presence are not enough for visual work; grounding, visibility, density, occlusion, and responsibility split must be reviewed.
- QIF is a living ledger, not a final report.

## Boundaries

This release does not claim:

- full automation of the new gates
- semantic truth
- complete UI comprehension validation
- QIF v0.3 executable verifier support
- Mission Control redesign

It claims that AOF's next operating baseline now includes pre-implementation stopping power and done-before evidence discipline.

## Next Frontier

After `v6.4.0`, the strongest next frontier is to implement the new gates as first-class runtime commands and schemas, then reflect them in Mission Control / Human Recognition surfaces.
