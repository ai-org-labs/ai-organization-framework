# AOF v6.1 Work Governance Integration Plan

## Purpose

`v6.1` integrates Work Governance as the next AOF frontier after `v6.0.0` Public Runtime Readiness.

AOF must evolve from a software-development-oriented runtime into a Human + AI + Tool Work Governance OS that can govern work across domains. The first slice is intentionally not a large implementation PR. It defines the decomposition, overlap analysis, schema integration plan, artifact naming plan, non-goals, and phased PR plan required before adding new runtime commands.

## Runtime Basis

- runtime status command: `organization-status --project .`
- roadmap command: `roadmap-status --project .`
- situation assessment artifact: `/tmp/aof-work-governance-situation.json`
- frontier task: `.aof/tasks/open/TASK-057.json`
- current release baseline: `v6.0.0`

Runtime reported `frontier-definition-needed` after `v6.0.0`, with no open implementation task. `TASK-057` is therefore the bounded direction task for this work.

## Release Claim

`v6.1` should make AOF able to describe, triage, assign, review, and decide work items as governed business work, not only software tasks.

The release claim is:

> AOF can represent a work item with explicit goal, reason, required skills, actor composition, Go / No-Go criteria, operational-map impact, council-ready output, external references, and bounded context.

## Existing Concept Overlap Analysis

### `task` vs `work item`

`task` remains the current runtime storage object. `work item` is the broader domain concept.

Integration rule:

- do not replace `.aof/tasks/` immediately
- add work-item fields and linked artifacts to task-backed work
- allow future non-task work sources through `external_refs`

### `operator brief` vs `governance brief`

`operator brief` remains the current visibility packet.

Integration rule:

- `governance brief` is the Work Governance summary layer
- it may be projected into operator brief / Mission Control
- it must not become an independent truth store

### PMO vs Work Governance

PMO is a view or alias for project-oriented teams.

Integration rule:

- AOF Core uses `Work Governance`
- Mission Control may show a PMO View
- artifacts should prefer `governance-*` or `work-*` names over `pmo-*`

### Skillful Actor vs Actor Composition

Skillful Actor proves that an actor can be assigned with skills, capability fit, resources, policies, and review gates.

Integration rule:

- actor composition determines which actor set is needed for a work item
- actor skill packet remains the assignment contract for each selected actor
- actor composition must record authority boundaries for human, AI, tool, and automation actors

### Need Validation vs Work Governance

Need Validation remains the pre-project authority gate.

Integration rule:

- Work Governance starts after intake and may request Need Validation
- raw Need must not become a project without Need Validation
- Work Governance may reject, defer, queue, or escalate work before project creation

### Mission Control vs Runtime Truth

Mission Control is projection only.

Integration rule:

- no independent Mission Control work status
- no independent priority
- no independent approval state
- no independent Go / No-Go judgment
- no independent QIF verdict

## Operational Map Requirements

Operational Map is the canonical structure showing how work flows, who participates, where decisions happen, what evidence is used, and which external systems are referenced.

Map categories:

- work-flow-map
- information-flow-map
- decision-approval-flow-map
- evidence-flow-map
- actor-role-tool-interaction-map
- external-reference-map
- system-architecture-map

Builder responsibility:

- check whether each work item changes any operational map
- update affected maps when structure changes
- record `no_change` with a reason when no map update is needed

Guardian responsibility:

- verify that the map matches reality
- block completion when map impact is missing or stale

## Go / No-Go Requirements

Every governed work item must be able to express:

- `go`
- `no_go`
- `conditional_go`
- `needs_human_approval`
- `needs_more_evidence`
- `needs_rework`

Go / No-Go visualization should be chosen by judgment need, not decoration:

- decision matrix
- dependency map
- risk heatmap
- operational flow
- actor responsibility map
- architecture / system map
- information flow map
- evidence coverage map
- readiness checklist

## Non-Goals

`v6.1` does not implement:

- Jira full bidirectional sync
- Mission Control independent judgment logic
- Mission Control as source of truth
- character UI
- sugoroku UI
- game-like visualization
- every external tool adapter
- fully autonomous Work Governance
- software-only PMO positioning
- perfect automatic diagram generation
- mandatory large diagrams for every work item

## Acceptance Criteria

`v6.1` is complete when:

- Work Governance is defined as the core concept, with PMO as a view or alias
- work item is defined as broader than task
- each governed work item can have a goal, success criteria, Go / No-Go criteria, required skills, required roles, and downstream dependency
- human, AI, tool, and automation actors can be represented with authority boundaries
- actor selection reason is recorded
- interrupts pass through Work Governance triage before changing the frontier
- current frontier protection is explicit
- human approval queue is representable
- Council-ready output is standardized
- Builder output includes operational-map impact
- Guardian verifies goal achievement, actor selection, visualization, evidence, and operational-map consistency
- Council can judge Go / No-Go from standardized outputs
- Context Pack keeps AI execution bounded
- External Reference model records relationship, sync policy, and source of truth
- Operational Map is defined as a core concept
- Mission Control remains a projection of runtime artifacts
- software and non-software examples exist

## Self-Review

This plan is consistent with the current AOF direction because it:

- preserves Need Validation as the pre-project gate
- preserves Skillful Actor as the assignment-level contract
- preserves Mission Control as visibility, not authority
- extends `task` into broader `work item` semantics without deleting current runtime storage
- makes Work Governance cross-domain instead of PMO-only
- adds stop points before large implementation

Primary risks:

- too many artifact families at once
- Work Governance may become documentation-heavy before CLI support exists
- Mission Control may drift toward UI-owned state if not guarded
- schema expansion may duplicate existing task, actor, and council fields

Mitigation:

- implement in phased PRs
- add narrow validators before dashboard expansion
- treat aliases as compatibility layers
- require artifacts to carry refs rather than copied raw bodies
