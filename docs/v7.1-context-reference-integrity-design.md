# AOF v7.1 Context and Reference Integrity Design

## Purpose

`v7.1.0` prevents AOF runtime work from depending on hidden conversation context,
undeclared source material, or stale external references.

`v7.0.0` made the agent session path reconstructable. That is necessary, but not
sufficient: a session can be fully reconstructable while still relying on facts,
files, screenshots, URLs, or assumptions that were never declared as runtime
inputs.

The bounded v7.1 claim is:

> AOF can identify which context and external references a work item relies on,
> verify that those inputs are declared and reviewable, and surface missing,
> stale, or hidden context as a blocker before Council approval or release
> sign-off.

## Human Wording

v7.1 should answer:

- What did the AI use as context?
- Which files, docs, screenshots, URLs, or external systems did it rely on?
- Are those inputs declared in artifacts rather than buried in chat memory?
- Are references current enough for the claim being made?
- Can Council see the inputs before approving the work?
- What remains unverified even if all references exist?

## Existing AOF Surface

AOF already has:

- `context-pack` writer and schema
- `external-ref` schema
- `work-governance-benchmark` checks for context pack compression and external
  source-of-truth / sync-policy linkage
- `agent-session-record` from v7.0, which links task, requirement, test,
  risk, decision, and release-ready evidence
- Mission Control projection from canonical artifacts

v7.1 should not duplicate those surfaces. It should add an integrity layer over
them.

## Core Concepts

### Context Dependency

A declared runtime input that may affect work output.

Examples:

- local file
- AOF artifact
- release note
- schema
- screenshot
- browser observation
- GitHub issue / PR
- external article / documentation
- prior decision record
- human instruction

### Context Pack Integrity

A context pack is integrity-ready only when:

- it summarizes state instead of embedding large raw bodies
- every important input has a resolvable `evidence_ref` or `external_ref`
- required context is connected to the work item or session
- missing context is explicitly represented as a blocker
- the pack states what is not proven by the context

### External Reference Integrity

An external reference is integrity-ready only when:

- source system and URL are declared
- source-of-truth boundary is explicit
- sync policy is explicit
- access or observation date is declared when freshness matters
- usage purpose is declared
- stale or unavailable reference can fail the audit
- reference presence is not treated as semantic truth

### Hidden Context

Hidden context is any material input used by the AI that is not represented in
the work item, context pack, external ref, session event, Council review, or
release evidence.

Hidden context must be represented as:

- `blocking` when it affects approval, release readiness, or user-facing claims
- `warning` when it is low-impact but should be declared before reuse
- `accepted_residual_risk` only when Council explicitly accepts the gap

## Artifact Design

### 1. `context-integrity-record`

Purpose:

- declare the context dependencies for one work item or session
- bind context refs to a specific claim, decision, or output
- record whether any hidden context remains

Required fields:

- `artifact_type`
- `record_id`
- `recorded_at`
- `work_item_id`
- `work_item_ref`
- `session_ref`
- `context_pack_refs`
- `declared_context_refs`
- `required_context_refs`
- `missing_context_refs`
- `hidden_context_signals`
- `integrity_status`
- `not_proven`
- `source_task_id`
- `source_parent_session_id`

Status values:

- `ready`
- `warning`
- `blocked`
- `accepted_residual_risk`

### 2. `external-reference-integrity-record`

Purpose:

- normalize freshness and availability checks for external refs without
  claiming semantic truth
- make stale or unavailable references visible to Council

Required fields:

- `artifact_type`
- `record_id`
- `recorded_at`
- `external_ref`
- `external_ref_artifact_ref`
- `source_system`
- `url`
- `relationship`
- `source_of_truth`
- `sync_policy`
- `usage_purpose`
- `freshness_required`
- `observed_at`
- `freshness_status`
- `availability_status`
- `integrity_status`
- `not_proven`
- `source_task_id`
- `source_parent_session_id`

Freshness values:

- `not_required`
- `current`
- `stale`
- `unknown`

Availability values:

- `available`
- `unavailable`
- `not_checked`

### 3. `context-reference-integrity-audit`

Purpose:

- release-grade verification command for v7.1
- narrow contract checks, not broad semantic scoring

Checks:

- implementation-grade tasks have context integrity records
- context records link to actual task/session refs
- required context refs resolve
- missing context is represented as blocking or accepted residual risk
- external refs used by context packs have integrity records
- stale external refs fail unless accepted by Council
- Council review evidence includes context/reference integrity refs
- session streams include reference-use or artifact-read events where available
- `not_proven` boundary is present

## Mission Control Projection

Mission Control should show:

- context integrity status: `ready / warning / blocked`
- missing context blockers
- stale external reference blockers
- top referenced files / artifacts / URLs
- Council-visible input summary
- latest context/reference audit result
- "not proven" statement

The viewer must not become source of truth. It only projects:

- context integrity records
- external reference integrity records
- audit output
- session stream links
- Council review links

## Council Gate

Council approval for implementation-grade work should require:

- context integrity record exists
- hidden context signals are empty, warning-only, or explicitly accepted
- external refs are current enough for the claim
- `not_proven` boundary is visible
- reviewer can trace the relevant context through artifacts, not conversation
  memory alone

## Release Gates

### Gate 1: Contract

- schemas exist for context integrity, external reference integrity, and audit
- writer commands create schema-valid artifacts
- audit command validates committed fixtures

### Gate 2: Negative Coverage

Audit must fail:

- missing context integrity record
- unresolved required context ref
- hidden context signal without blocker/acceptance
- stale external ref used for a freshness-sensitive claim
- Council review with no context/reference evidence
- missing `not_proven` boundary

### Gate 3: Continuity

Existing gates remain green:

- session observability audit
- work readiness audit
- quality ledger audit
- Archmap impact audit
- review provenance audit
- evidence independence audit
- organization verification
- command routing audit
- CLI help benchmark
- focused tests
- `npm test`
- `npm run smoke`

## Boundary

v7.1 does not prove that a referenced source is true.

It proves only:

- the source was declared
- the reference resolves or its failure is visible
- freshness expectations are represented
- Council can see the input chain
- hidden context is treated as a governance risk

Semantic correctness still requires QIF, expert review, reproduction tests,
operator validation, or outcome feedback.

## Design Decision

Proceed with v7.1 as an integrity layer over existing Work Governance and v7.0
session observability.

Do not make v7.1 a broad research or web-verification system. Keep the first
slice local, artifact-backed, and release-gateable.
