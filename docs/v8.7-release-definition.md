# AOF v8.7 Release Definition

## Theme

`v8.7.0` is the Approval Authenticity and Adapter Capability Alignment release.

The release fixes the gap found after `v8.6.0`: an external-write approval must
not pass merely because an approval-shaped record exists. It must prove that the
adapter can perform the requested operation, that a real human approval is
recorded, and that the approval is bound to a concrete provider operation target
with credential, budget, rollback, and stop-condition boundaries.

## What v8.7 Adds

- `provider-operation-target-record` for concrete provider/resource/operation
  targets
- `human-approval-record` for explicit human approval evidence
- structured `target_operation_ref`, `credential_scope`, `budget`, and
  `rollback` fields on provider execution approval records
- provider execution approval audit checks for adapter operation alignment,
  active human approval, target binding, payload-scope alignment, bounded
  credential scope, budget, and rollback readiness
- negative regression coverage for read-only adapter overclaims
- corrected v8.6 self-hosting preflight evidence that is now blocked/read-only
  on main

## Release Boundary

This release does not execute a production provider write. The v8.7 self-hosting
record is a governed preflight that proves whether a later external write claim
is properly bounded before execution.

Not proven:

- live provider execution success
- provider output correctness
- credential safety beyond declared scope
- billing safety beyond declared budget
- rollback execution success
- autonomous external execution

## Required Gates

- provider execution approval audit passes with one blocked v8.6 correction and
  one bounded v8.7 preflight approval
- read-only adapter external-write overclaim fails in regression coverage
- active human approval record resolves and matches the target payload scope
- target operation record resolves to concrete provider/resource/operation
  metadata
- release-state audit remains green without treating production safety as proven
- Mission Control and release notes state the preflight boundary clearly

## Release Decision

Ship only if AOF makes it harder to confuse:

- Council review with independent human approval
- read-only adapter capability with write capability
- abstract target text with concrete provider operation scope
- approval with production execution
- green audit with real-world provider safety
