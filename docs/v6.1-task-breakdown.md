# AOF v6.1 Task Breakdown

## Purpose

This document fixes the staged task plan for `v6.1.0` Work Governance Integration.

`v6.1` must not become a single large undifferentiated implementation. It proceeds through bounded AOF work items, each with a clear handoff to the next stage.

## Task Sequence

### TASK-057: Define v6.1 Work Governance Integration Plan

Goal:

- Establish the Work Governance direction and integration boundaries.

Required outputs:

- decomposition plan
- existing concept overlap analysis
- schema integration plan
- artifact naming plan
- non-goals
- phased PR plan
- core model update
- roadmap/release-plan update

Go / No-Go:

- Go if Work Governance is clearly broader than PMO and Mission Control remains projection-only.
- No-Go if the plan weakens Need Validation, duplicates existing runtime concepts, or makes the dashboard a source of truth.

Handoff:

- TASK-058 receives the schema and fixture implementation scope.

### TASK-058: Add Work Governance Schemas And Fixtures

Goal:

- Add the minimal schema surface and fixtures required to represent governed work items.

Required outputs:

- `schemas/aof-work-item-goal.schema.json`
- `schemas/aof-actor-composition.schema.json`
- `schemas/aof-council-ready-output.schema.json`
- `schemas/aof-go-no-go-visualization.schema.json`
- `schemas/aof-operational-map-change-log.schema.json`
- `schemas/aof-context-pack.schema.json`
- `schemas/aof-external-ref.schema.json`
- software-domain fixture
- non-software-domain fixture
- schema validation tests

Go / No-Go:

- Go if fixtures validate and do not duplicate current task, actor skill, or council review truth.
- No-Go if schemas make Work Governance software-only or let external refs omit source-of-truth / sync policy.

Handoff:

- TASK-059 receives stable schema contracts for CLI writers.

### TASK-059: Add Narrow Work Governance CLI Writers

Goal:

- Add narrow write commands that produce schema-valid Work Governance artifacts.

Required outputs:

- `work-item-goal`
- `actor-compose`
- `council-ready-output`
- `go-no-go-visualization`
- `operational-map-change-log`
- `context-pack`

Go / No-Go:

- Go if commands fail clearly on missing required goal, skill, actor, Go/No-Go, map, or council-ready fields.
- No-Go if commands silently create incomplete governance artifacts.

Handoff:

- TASK-060 receives CLI-produced artifacts for audit and benchmark coverage.

### TASK-060: Add Work Governance Audits And Benchmarks

Goal:

- Make Work Governance falsifiable.

Required outputs:

- `context-audit`
- `external-ref-audit`
- `operational-map-audit`
- `work-governance-benchmark`
- negative fixtures for stale context, missing actor selection, missing map impact, unresolved external refs, and missing council-ready output

Go / No-Go:

- Go if the validator catches weak or incomplete Work Governance artifacts.
- No-Go if test pass only proves JSON shape while missing semantic guardrails.

Handoff:

- TASK-061 receives audited artifact state for Mission Control projection.

### TASK-061: Project Work Governance Into Mission Control

Goal:

- Show Work Governance state in Mission Control without making the viewer a source of truth.

Required outputs:

- Work Governance dashboard projection
- work board projection
- Go / No-Go panel
- actor composition panel
- approval queue projection
- interrupt inbox projection
- external reference panel
- operational map refs panel

Go / No-Go:

- Go if every displayed value is derived from canonical runtime artifacts.
- No-Go if Mission Control stores independent status, priority, approval, QIF, map, or Go/No-Go truth.

Handoff:

- TASK-062 receives the product surface for cross-domain examples.

### TASK-062: Add Cross-Domain Work Governance Examples

Goal:

- Prove Work Governance is not software-only.

Required outputs:

- one software example
- one non-software example
- `docs/v6.1-work-governance-examples.md` explaining source-of-truth, sync policy, actor composition, Operational Map impact, and Council-ready output
- quickstart update if needed

Go / No-Go:

- Go if a first-time operator can understand both examples without reading raw schema docs.
- No-Go if examples look like empty folders or software-only PMO templates.

Handoff:

- TASK-063 receives release evidence and documentation.

### TASK-063: Close v6.1.0 Work Governance Release

Goal:

- Release `v6.1.0` with honest claims and reproducible evidence.

Required outputs:

- `docs/v6.1-release-definition.md`
- `docs/v6.1-release-checklist.md`
- `docs/v6.1.0-release-notes.md`
- package/bootstrap/release-state alignment
- runtime verification evidence
- tag and GitHub release readiness

Go / No-Go:

- Go if Work Governance contracts, CLI writers, audits, examples, docs, and Mission Control projection are consistent.
- No-Go if v6.1 claims autonomous Work Governance, full external sync, or dashboard-owned truth.

## Current Next Step

Complete TASK-062 by validating the adoption-facing example document against the committed software and manufacturing Work Governance artifact chains, then hand off to TASK-063 for release closure.
