# AOF v6.7.0 Release Notes

`v6.7.0` is the Verifiable Governance release.

It follows `v6.6.0` Archmap-Aware Mission Control. v6.6 made architecture impact visible and Council-disposed. v6.7 turns that pattern into machine-checkable release governance.

## What Changed Since v6.6.0

- Added `archmap-impact-audit`.
- Added `review-provenance-audit`.
- Added `evidence-independence-audit`.
- Integrated all three audits into `release-state-audit`.
- Added release-state audit schema coverage for `governance_audits`.
- Strengthened `cli-help-benchmark` so command help fails when inputs or outputs are missing.
- Added default AI-readable input/output hints for commands without explicit hint definitions.
- Fixed Mission Control / visibility Archmap latest work selection so later task ids are not hidden by older `recorded_at` timestamps.
- Updated `docs/archmaps/aof-runtime-current.archmap` with Verifiable Governance and release sign-off nodes.
- Closed `TASK-072` through `TASK-075` with Council review evidence and opened `TASK-076` for release closure.

## Human-Facing Meaning

v6.7 says:

> AOF must not merely say governance happened. It must be able to show, through runtime commands, that release-critical work has architecture impact disposition, Council review provenance, and evidence that is not only maker-authored or self-attested.

For an operator, this means release readiness now asks:

- did the work declare architecture impact?
- did Council actually review the completed work?
- does the review point to resolvable evidence?
- is at least some evidence independent of the maker's own claim?
- does release-state-audit consume these checks before sign-off?
- can AI read command help without guessing inputs and outputs?

## New Operating Rules

- `release-state-audit` is no longer only release-surface drift detection.
- Release sign-off now includes v6.7 governance audit gates.
- Done implementation work after `TASK-071` requires Council review provenance.
- Done implementation work after `TASK-071` requires at least one independent evidence category.
- Command help must expose inputs and outputs, or `cli-help-benchmark` fails.
- Mission Control latest Archmap work must follow task progression when task ids are present.

## Boundaries

This release does not claim:

- that machine checks prove semantic truth
- that Council judgment is always correct
- that independent evidence is sufficient by itself
- full external QIF verifier execution
- Node.js 25+ support

It claims that AOF now has a stronger release-readiness structure: governance claims are checked through runtime commands before sign-off, and missing governance evidence is harder to overlook.

## Next Frontier

After `v6.7.0`, the strongest next frontier remains bounded by the roadmap: improve runtime planning and evidence discipline without adding unbounded automation. The next slice should be selected through a runtime-backed direction review after this release is tagged.
