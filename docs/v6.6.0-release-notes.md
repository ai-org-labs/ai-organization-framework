# AOF v6.6.0 Release Notes

`v6.6.0` is the Archmap-Aware Mission Control release.

It follows `v6.5.0` Execution Hygiene and Command Safety. v6.5 made AOF runs smaller and safer. v6.6 addresses the next governance gap: implementation work can change organization/runtime architecture without a first-class map-impact decision.

## Human-Facing Meaning

v6.6 says:

> If AOF changes the shape of the organization or runtime, the architecture map impact must be visible, reviewed, and traceable.

For each implementation-grade work item, AOF should make clear:

- whether architecture changed
- which map source proves the current architecture view
- which node, edge, zone, or boundary changed
- why the change matters
- whether Council approved, rejected, or deferred the map impact
- whether Mission Control is projecting canonical evidence rather than inventing truth

## New Operating Rules

- Architecture impact is part of governed work closure.
- Work items should declare `archmap_update_required`, `archmap_unaffected`, or `archmap_deferred_with_reason`.
- Archmap source files and AOF artifacts are source of truth.
- Mission Control is a projection surface only.
- Icons and diagrams improve recognition but do not prove quality.
- Node.js 22 and Node.js 24 are the supported CI runtime lanes.
- Node.js 25+ remains outside the supported runtime lane.

## Boundaries

This release does not claim:

- full Archmap renderer integration into Mission Control
- required `@archmap/core` or `@archmap/icons` runtime dependencies
- externally validated visual comprehension
- architecture quality proven by diagram existence
- executable QIF v0.3+ verifier replacement support
- Node.js 25+ support

It claims that AOF now has a bounded architecture-impact governance layer that can connect work items, Archmap source, Council review, and Mission Control projection.

## Next Frontier

After `v6.6.0`, the strongest next frontier is to make the architecture impact contract machine-verifiable for every implementation-grade work item, so missing map-impact decisions are detected before release sign-off rather than reviewed manually.
