# AOF v9.4 Release Definition

## Theme

Provider-backed Controlled Execution Candidate.

## What The User Can Do

Users can inspect one controlled candidate record and see whether a provider-backed action has the required upstream evidence before any external side effect is attempted.

Before v9.4, the evidence existed across approval, target, reproduction, rollback, outcome, learning, operator acceptance, product value, and production-boundary artifacts. The operator still had to mentally assemble the chain.

After v9.4, AOF can create and audit a single controlled execution candidate that says:

- what provider-backed action is being considered
- which target operation it is bound to
- which approval, rollback, outcome, learning, and operator evidence supports it
- whether external write is only a bounded candidate
- whether production execution is explicitly not authorized
- what must stop the run

## Required Runtime Surface

- `provider-controlled-execution-candidate-record`
- `provider-controlled-execution-candidate-audit`
- `release-state-audit` includes the candidate audit for `9.4.0` and later

## Release Boundary

v9.4 does not authorize production provider execution.

It proves a controlled execution candidate can be represented and audited from canonical runtime artifacts.

## Acceptance Gates

- controlled execution candidate schema validates
- controlled execution candidate audit passes
- release-state audit includes the v9.4 gate
- product capability matrix explains the v9.4 capability delta
- release notes explain user value before mechanism
- full test and smoke suite pass
