# AOF v6.2 Release Definition

## Release

- version: `6.2.0`
- tag: `v6.2.0`
- theme: Worked Adoption Proof
- parent task: `TASK-064`

## Release Claim

`v6.2.0` proves that AOF can be introduced into a fresh managed project and guide a first-time operator to the first governed work item.

The release claim is intentionally bounded:

> AOF can initialize a fresh managed project, create a first governed work artifact chain, verify that chain, and explain the work in plain language: what the work is, why it matters, who judges it, what evidence exists, what is blocked, and what should happen next.

## Required Outcomes

- A fresh managed project can be initialized with `aof init --topology managed-project`.
- The fresh project receives the canonical `.aof/` bootstrap and orientation artifacts.
- A first governed work item chain can exist in that project:
  - work item goal
  - actor composition
  - Council-ready output
  - Go / No-Go visualization
  - Operational Map change log
  - Context Pack
  - External Reference
- `work-governance-benchmark` passes against the fresh-project chain.
- A human-readable recognition summary exists for the first governed work item.
- The recognition summary explains:
  - what the work is
  - why it matters
  - who judges it
  - what evidence exists
  - what is blocked
  - what should happen next
- No `.DS_Store` or legacy personal account reference is present in the fresh proof.
- The proof is reproducible through `adoption-proof-benchmark`.

## Authority Boundary

The release does not claim:

- that every external project will be semantically successful
- that AOF has full autonomous Work Governance
- that external tool synchronization is complete
- that the current viewer is a finished Human Recognition UX
- that self-hosting evidence alone proves public adoption

The release does claim:

- a clean managed-project adoption path is executable
- the first governed work item is structurally verifiable
- human-readable recognition output is part of the runtime contract
- correct JSON without operator recognition is incomplete

## Gates

1. Direction gate: runtime-backed post-v6.1 review selects Worked Adoption Proof as the next frontier.
2. Fresh setup gate: `adoption-proof-benchmark` initializes a managed project from scratch.
3. First work gate: a governed work item chain exists in the fresh project.
4. Verification gate: Work Governance checks pass against that fresh-project chain.
5. Recognition gate: the first governed work item has a plain-language recognition summary.
6. Hygiene gate: no `.DS_Store` or legacy personal identity references appear in the proof.
7. Release gate: package/bootstrap/active release state align with `6.2.0`, verification evidence is green, and `TASK-064` is approved.
