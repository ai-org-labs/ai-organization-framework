# AOF v6.5.0 Release Notes

`v6.5.0` is the Execution Hygiene and Command Safety release.

It follows `v6.4.0` Pre-Implementation Quality Gate. v6.4 made AOF harder to use as a post-hoc archive of wrong work. v6.5 addresses the next operational failure mode: AOF itself can become too large, too sticky, too tool-heavy, or too approval-fatiguing unless each run has a small work boundary and an explicit permission contract.

## Human-Facing Meaning

v6.5 says:

> AOF should not become a giant instruction blob or a permission-fatigue machine.

For each run, AOF should make clear:

- what work item is active
- what context is persistent
- what skills and artifacts are loaded on demand
- what local commands are safe to batch
- what writes are limited to `.aof`
- what project changes require care
- what external or dangerous operations require explicit approval
- what handoff is ready for Council judgment

## New Operating Rules

- CLI help is command routing, not runtime proof.
- Runtime claims require executed runtime evidence.
- Command category and command safety are separate axes.
- `safe_read` and `safe_local_write` are preapproved in normal local runs.
- `project_write`, `external_write`, and `dangerous` require per-run approval.
- Safe local command batching is encouraged to reduce permission fatigue.
- Work item boundaries and handoff artifacts should prevent long conversation memory from becoming the hidden source of truth.

## Boundaries

This release does not claim:

- automatic permission bypass
- external-write autonomy
- deploy/publish approval
- secrets or billing access
- production mutation safety
- full Mission Control redesign
- perfect command classification

It claims that AOF now has a first-class local execution hygiene contract that can separate safe runtime work from work requiring explicit human approval.

## Next Frontier

After `v6.5.0`, the strongest next frontier is to make Mission Control show the Execution Hygiene state directly: active work item, run contract, command safety boundary, actor handoff, Council readiness, blockers, and runtime proof.
