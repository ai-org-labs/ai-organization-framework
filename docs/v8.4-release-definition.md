# AOF v8.4 Release Definition

## Theme

Externalized Runtime Operator Safety Proof.

`v8.4.0` makes externalized runtime safety visible as a first-class
Mission Control decision surface. The operator should be able to see permission,
approval, provenance, freshness, risk, and not-proven boundaries before any
externalized execution bridge advances.

## Required Outcomes

- Mission Control projects `external_runtime_safety_projection` from canonical
  runtime artifacts.
- Safety states are explicit: `safe`, `blocked`, `needs_approval`, `stale`, and
  `not_proven`.
- Visible safety state links to release refs, work item refs, provider refs,
  resource refs, evidence refs, and governance action.
- Missing externalization, resource, provider, or freshness evidence remains
  visible as incomplete rather than silently passing.
- v8.3 operator acceptance, v8.2 operator validation, v8.1 provider adapter,
  v8.0 external resource, and v7.9 externalization readiness baselines remain
  compatible.

## Non-Goals

- This release does not execute external providers autonomously.
- This release does not grant production-write authority.
- This release does not manage credentials, billing, deploys, or hosted runtime.
- This release does not prove semantic correctness of external tool output.
- This release does not replace human or governance approval for risky actions.

## Acceptance Boundary

AOF may claim externalized runtime safety visibility only when Mission Control
derives the claim from canonical runtime artifacts and preserves the not-proven
boundary.

The `safe` status means safe for bounded operator review and planning, not
permission for autonomous external writes.
