# AOF v7.0 Direction Review: Agent Session Observability

## Runtime Context

- current shipped release: `v6.9.0`
- current runtime stage: `frontier-definition-needed`
- direction task: `TASK-084`
- source session: `SESS-POST-V69-STREAMING-DIRECTION`

`v6.9.0` made implementation readiness executable before work starts. The next unresolved gap is that AOF can verify artifacts after they exist, but it does not yet treat the live agent session itself as a first-class evidence stream.

## External Signal

GitHub announced public-preview Copilot agent session streaming on 2026-07-02:

- source: `https://github.blog/changelog/2026-07-02-copilot-agent-session-streaming-is-now-in-public-preview/`
- relevant signal: enterprise operators can access agent session activity such as prompts, responses, and tool calls across Copilot clients.
- available surfaces: streaming endpoint and REST API for recent usage records.
- governance implication: agent behavior is becoming an observable operational surface, not only a post-hoc artifact trail.

## AOF Interpretation

This does not mean AOF should depend on GitHub Copilot Enterprise features.

It does mean AOF should absorb the pattern:

> AI organization work should be inspectable as a session event stream: prompt, response, tool call, artifact write, decision, review, stop condition, and handoff.

Current AOF records many artifacts, but the operator still reconstructs the live work path by reading final files. That is weaker than an audit stream.

## Direction Candidate

`v7.0.0` should be considered for:

**Agent Session Observability and Runtime Event Stream**

The bounded v7 claim would be:

> AOF can record, normalize, and audit runtime session events so human operators can reconstruct who asked what, which tools ran, which artifacts changed, which decision was made, and why the loop stopped.

## Required Concepts

### Session Event Record

Canonical event unit for:

- user prompt
- orchestrator decision
- role assignment
- actor output
- tool call
- artifact write
- council review
- verification result
- blocker
- stop/defer/reopen decision

### Runtime Event Stream

Append-only ordered stream for one work item or session.

It should answer:

- what happened
- when it happened
- who/what initiated it
- which artifact changed
- which command/tool was invoked
- what evidence was produced
- what should happen next

### Session Observability Audit

Audit that checks:

- every implementation-grade task has a session stream ref
- every council decision has preceding evidence events
- every artifact write has a producing event
- every tool call has safety level and approval policy
- every loop has goal, verification, and stop condition events
- missing streamed evidence can be represented as blocking

## Relation To Existing AOF

This extends, not replaces:

- `execution-lineage`: lineage remains the artifact graph summary.
- `runtime-execution`: visibility remains the current runtime-backed answer surface.
- `quality-ledger`: quality-relevant changes remain semantic governance evidence.
- `work-readiness`: readiness still gates work before execution.
- `release-state-audit`: release audit should eventually consume session observability.

## Non-Goals

v7 should not require:

- GitHub Enterprise Cloud
- Copilot-specific APIs
- vendor-specific streaming
- live network streaming in the first implementation
- SIEM integration

The first implementation can be local and file-backed.

## Recommended v7 Slice

Start with a local append-only runtime event stream:

1. define `session-event-record` schema
2. define `runtime-event-stream` schema
3. add a narrow writer command
4. add `session-observability-audit`
5. connect existing runtime commands to record minimal events
6. expose the current session stream in Mission Control / operator surfaces

## Council Judgment

Adopt as the strongest post-v6.9 frontier candidate.

Reason:

- It directly addresses the user's recurring concern that AOF claims progress but humans cannot see the live work path.
- It aligns with market movement toward auditable agent session telemetry.
- It is vendor-neutral if implemented as a local canonical event stream first.
- It strengthens runtime-backed answers without overclaiming semantic truth.

## Boundary

Session observability proves reconstructability of process.

It does not prove:

- the answer is correct
- the work had market value
- the agent made good judgments
- the UI is comprehensible
- the operator accepted the outcome

Those still require QIF, Guardian/Council review, human/operator validation, and outcome feedback.
