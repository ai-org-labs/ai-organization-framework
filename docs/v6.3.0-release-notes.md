# AOF v6.3.0 Release Notes

`v6.3.0` is the AI Command Help Surface release with QIF v0.3.0 provider alignment.

It follows `v6.2.0` Worked Adoption Proof. v6.2 proved that a fresh project can reach first governed work. v6.3 improves the AI operator path: before an AI can govern work well, it must be able to cheaply discover the right runtime command and understand what command success or failure means.

## What Changed Since v6.2.0

- Added registry-backed AI command help.
- Added `aof --help --json` as a compact command index.
- Added `aof <command> --help --json` as structured command-level help.
- Added failure meaning and QIF boundary language to command help.
- Added `cli-help-benchmark`.
- Registered `cli-help-benchmark` in the command catalog and command registry.
- Updated the active QIF provider profile from QIF `v0.2.1` to QIF `v0.3.0`.
- Recorded QIF Core, QIF Discovery Layer, and AOF Integration as provider layers.
- Updated QIF provider and quality definition docs to keep the semantic truth boundary explicit.
- Added `docs/v6.3-release-definition.md`.
- Added `docs/v6.3-release-checklist.md`.

## Human-Facing Meaning

v6.3 makes AOF easier for AI to operate.

Instead of forcing an AI orchestrator to read a large CLI reference or guess from memory, AOF can now answer:

- what commands exist
- which commands are top-level routing commands
- what a command is for
- what inputs and outputs are expected
- what failure means
- what QIF quality boundary applies

This is not a human UX redesign. It is an AI transmission-efficiency layer.

## QIF v0.3.0 Boundary

QIF v0.3.0 adds the Discovery Layer design boundary: Quality Intents should be discovered from concerns, stakeholders, potential harms, and loss boundaries before evidence is judged.

AOF now records QIF v0.3.0 as the active external provider profile.

However, QIF v0.3.0 does not yet provide replacement schemas, example packages, or verifier rules. AOF therefore keeps its active machine-readable package v0.2.1-compatible until QIF publishes executable v0.3 artifacts.

## Boundaries

This release does not claim:

- command semantics are proven merely because help exists
- benchmark pass proves semantic quality truth
- QIF v0.3.0 has executable verifier support inside AOF
- all benchmark outputs are now fully human-explained
- Mission Control UX is solved

It claims only that AOF now has a compact, structured, registry-backed command help surface for AI runtime operation and an explicit QIF v0.3.0 provider alignment.

## Next Frontier

After `v6.3.0`, the strongest next frontier is QIF-Governed Benchmark Explanation.

The reason is direct: command help can now expose QIF boundary language, but benchmark outputs still need a first-class explanation layer that tells humans what was checked, what standard was expected, what failure would mean, what remains unproven, and when governance must escalate.
