# AOF v6.0 Public Runtime Readiness

## Purpose

`v6.0.0` is the public adoption readiness release for the historyless `ai-org-labs/ai-organization-framework` baseline.

The release does not claim a new autonomy layer. It makes the current runtime safer to adopt by proving what is runnable, what is only documented, what the examples contain, what the viewer can show, and what remains blocked by external GitHub credentials.

Readiness levels:

| Level | Meaning | Evidence |
|---|---|---|
| registry-routable | command exists in the command registry and has category / purpose metadata | `.aof/command-registry.json`, `command-register` |
| routing-audited | top command routing and category coverage pass | `command-routing-audit` |
| runtime-verified | organization / release / decision surfaces validate | `organization-verify`, `decision-verify`, `release-state-audit` |
| test-covered | automated tests exercise the command or its runtime path | `npm test` |
| smoke-covered | end-to-end mock-provider flow reaches approval/outcome paths | `npm run smoke` |
| live-provider-dependent | command needs external model/provider credentials for live execution | `council-exec --invoke-model`, `live-verify` with non-mock provider |

Important boundary:

`v6.0.0` does not claim that every command has been manually executed with every production argument shape. It claims that the command surface is registered, routable, schema-validated where applicable, covered by the current test/smoke suite, and bounded by explicit provider/runtime constraints.

## Example Readiness

Examples are starter templates, not completed projects.

Included examples:

- `examples/aidlc-template`: AIDLC-oriented runtime template with actors, workflow, seed sessions, signal, decision template, and governance files.
- `examples/generic-template`: non-AIDLC service-design template with actors, workflow, signal, decision template, and governance files.

The `.aof/artifacts/` directories intentionally start empty. Runtime artifacts are generated by quickstart commands and one-shot verification runs. This is preferable to shipping stale generated artifacts as if they were universal examples.

The public readiness requirement is therefore:

- seed configuration is present
- sample sessions/signals exist where useful
- quickstart can generate fresh artifacts
- tests prove the templates load and run through runtime paths

## Viewer State

The current viewer is a runtime-backed AOF Mission Control Dashboard.

It can display:

- top mission tiles for current mission, release/frontier, runtime-backed status, risk/blocker count, and next recommended action
- a central AOF Kanban Board with Backlog, Discovery, Need Validation, Planning, Role Work, Council Review, and Approved / Done lanes
- right-side overall summary with active loops, open tasks, blockers, current frontier branch, and key risks
- lower task flow, blocked task, role workload, and evidence/proof coverage panels

It should not be described as a finished full Human Recognition UX. The viewer remains useful as a read-only projection of canonical runtime artifacts, but the primary operator truth is still the JSON-backed runtime surfaces:

- `operator-brief`
- `operator-progress`
- `tree-position`
- `evidence-drill-down`
- `runtime-execution`

## Public Migration Boundary

The public baseline must not contain:

- legacy personal GitHub handles
- legacy personal email addresses
- old personal GitHub URLs or historical CI URLs
- copied `.git` history
- `.DS_Store`

The public repo remote is:

- `https://github.com/ai-org-labs/ai-organization-framework.git`

GitHub push/release is external-state dependent. If credentials still authenticate as an account without write access to `ai-org-labs`, local release/tag can be prepared but remote release cannot be completed.

## Release Gates

`v6.0.0` is releasable only if:

- QIF package validation passes
- organization verification passes
- decision verification passes
- command routing audit passes
- release-state audit passes after v6 manifest refresh
- `npm test` passes
- `npm run smoke` passes
- quickstart points at `v6.0.0`
- README points at v6 release docs
- examples are documented as starter templates
- Mission Control Dashboard is visible and viewer limitations are explicit
- personal account scans are clean
- `.DS_Store` scans are clean

## Non-Goals

`v6.0.0` does not:

- add a new autonomous backend
- complete the full character-based Human Recognition Interface
- claim every command was live-provider executed
- claim semantic market truth
- weaken Need Validation, QIF, or Skillful Actor Runtime gates
