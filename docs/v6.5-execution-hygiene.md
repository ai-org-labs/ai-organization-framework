# AOF v6.5 Execution Hygiene

## Purpose

`v6.5` introduces Execution Hygiene as the operating discipline that keeps AOF usable across long-running AI work without turning every run into a giant prompt, giant `CLAUDE.md`, tool sprawl, or permission-fatigue loop.

The core idea:

> AOF should preserve only the minimum persistent context required for the current work item, load skills and artifacts on demand, keep work small, and make handoff/council evidence explicit before execution expands.

## Hygiene Principles

1. Minimal persistent context: keep durable context small and artifact-backed.
2. Skills / artifacts on demand: load specialist knowledge only when the current work item requires it.
3. Work item session boundary: every non-trivial run should have a bounded work item and handoff surface.
4. Small diff / small work item: prefer small accepted changes over broad speculative edits.
5. Specialist actor composition: assign the smallest useful actor set for the current problem.
6. Council-ready handoff: actor output should be summarized into a decision-ready packet.
7. Approved run contract: each run should state what is preapproved and what needs per-run approval.
8. Batch safe local commands: group safe local reads/verifications when it reduces operator fatigue.
9. Permission fatigue avoidance: do not ask for approval repeatedly for safe local actions already covered by the run contract.

## Failure Modes Addressed

### Giant Persistent Context

Risk:

- AOF project instructions or persistent context grow until every run carries stale or irrelevant material.

Rule:

- Put details into artifacts and context packs, not permanent prompt state.
- Keep persistent instructions to routing, authority, and safety boundaries.

### One Huge Instruction

Risk:

- A single integrated instruction tries to carry mission, design, execution, review, release, and tool policy all at once.

Rule:

- Split work into Work Item Goal, Context Pack, Actor Composition, Council-ready Output, and Go / No-Go surfaces.

### Mid-Run Compaction Loss

Risk:

- Long sessions compact away critical rationale and the next actor cannot reconstruct the work.

Rule:

- Use work item session boundaries and handoff artifacts before context becomes fragile.

### Tool Sprawl

Risk:

- MCPs, external refs, plugins, and connectors are all loaded or invoked by default.

Rule:

- External tools must be selected by work item need, capability fit, and approval boundary.

### Design Dumping

Risk:

- The operator hands vague strategy to an AI and receives unbounded design output that cannot be judged.

Rule:

- Work Item Goal, Go / No-Go, and Council-ready Output are required for non-trivial design work.

### Weak Review

Risk:

- Builder output is reviewed by the same assumptions that created it.

Rule:

- Guardian / QIF review must be separate from Builder output, and specialist actors should be selected when the risk is domain-specific.

### Conversation Drag

Risk:

- Old context from prior work items silently biases the next run.

Rule:

- Use work item handoff artifacts instead of relying on the same conversation memory.

## Command Safety Levels

`v6.5` treats command category and command safety as separate axes.

Command category answers:

- What kind of command is this? `read`, `verify`, `write`, `execute`, or `observe`.

Command safety level answers:

- What permission boundary applies to this command in a normal run?

Safety levels:

- `safe_read`: state inspection, brief generation, context-pack reading, audit display, verification output.
- `safe_local_write`: `.aof/` artifact updates, goal updates, logs, visibility packets, local benchmark evidence.
- `project_write`: project docs, schema, source, test, example, or non-`.aof` repository updates.
- `external_write`: GitHub / Jira / Slack / API / model-provider / other external service writes or credentialed/billable calls.
- `dangerous`: delete, deploy, publish, billing, secrets, production, or irreversible operation.

## Approved Run Contract

A standard local AOF run may preapprove:

- `.aof/` artifact updates
- local AOF runtime artifact updates
- read commands
- verify / audit commands
- agreed-with-human goal updates
- batched safe local commands

Per-run human approval is required for:

- `git push`
- package publish
- deploy
- external service writes
- secrets access
- billing operations
- destructive operations
- production mutations
- irreversible operations

## Runtime Contract

Execution Hygiene is not a decorative guideline. It must be visible through runtime evidence:

- command registry entries expose `safety_level`
- command help exposes `safety_level` and `approval_policy`
- CLI help benchmark verifies safety metadata presence
- runtime claims still require executed verification, audit, self-audit, or retrospective evidence

## Boundary

This document does not authorize AOF to bypass tool or platform permissions. It defines the AOF-side classification and run contract so that safe local actions can proceed without repeated human interruption, while external or irreversible actions remain explicitly gated.
