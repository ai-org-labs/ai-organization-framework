# AOF v6.1 Release Definition

## Release

- version: `6.1.0`
- tag: `v6.1.0`
- theme: Work Governance Integration
- parent tasks: `TASK-057` through `TASK-063`

## Release Claim

`v6.1.0` makes AOF able to represent governed work as cross-domain Human + AI + Tool work, not only software implementation tasks.

The release claim is intentionally bounded:

> AOF can describe, write, verify, and project a governed work item with goal, reason, actor composition, authority boundaries, Council-ready output, Go / No-Go state, Operational Map impact, bounded context, and external source-of-truth refs.

## Required Outcomes

- Work Governance is defined as a core AOF concept.
- Work item is defined as broader than task while `.aof/tasks/` remains the current runtime task store.
- Work Governance schema contracts exist for:
  - work item goal
  - actor composition
  - Council-ready output
  - Go / No-Go visualization
  - Operational Map change log
  - Context Pack
  - External Reference
- Narrow CLI writers can persist schema-valid Work Governance artifacts.
- `work-governance-benchmark` verifies the committed fixture chains.
- Mission Control projects Work Governance state from canonical artifacts without becoming a source of truth.
- Software and non-software examples are documented for first-time operators.
- Roadmap and release plan describe `v6.1.0` honestly.

## Authority Boundary

The release does not claim:

- fully autonomous Work Governance
- full Jira, QMS, CRM, or other external-tool synchronization
- replacement of `.aof/tasks/` with a complete work-item store
- semantic truth of fixture scenarios
- that Mission Control owns priority, approval, map, or Go / No-Go state
- that every proposed audit command from the early direction plan is fully productized

The release does claim:

- the Work Governance artifact chain is structurally defined
- software and non-software work can share the same governed chain
- AI, human, and tool actors can be represented with authority boundaries
- Council-ready output and Go / No-Go state can be verified from artifacts
- external systems can be named with source-of-truth and sync-policy semantics
- Mission Control can display Work Governance as projection only

## Gates

1. Direction gate: `TASK-057` establishes Work Governance boundaries and non-goals.
2. Schema gate: `TASK-058` schemas and fixtures validate across software and non-software domains.
3. CLI gate: `TASK-059` writer commands persist schema-valid Work Governance artifacts.
4. Verification gate: `TASK-060` benchmark catches incomplete Work Governance chains.
5. Projection gate: `TASK-061` Mission Control displays Work Governance without owning truth.
6. Example gate: `TASK-062` examples explain both fixture chains to first-time operators.
7. Release gate: package/bootstrap/active release state align with `6.1.0`, verification evidence is green, and `TASK-063` is approved.
