# AOF v6.5 Release Definition

## Release

- version: `6.5.0`
- tag: `v6.5.0`
- theme: Execution Hygiene and Command Safety
- parent task: `TASK-070`

## Release Claim

`v6.5.0` makes AOF safer to operate in real AI development sessions.

The bounded claim is:

> AOF can distinguish command discovery from runtime proof, classify command safety, preserve a minimal run contract, and make the next work item frontier explicit without turning each session into a giant prompt or permission-fatigue loop.

## Required Outcomes

- Execution Hygiene is documented in `docs/v6.5-execution-hygiene.md`.
- CLI help remains an AI-oriented routing surface, not proof that runtime work happened.
- Command registry entries expose `safety_level` and `approval_policy`.
- Command help exposes safety metadata for AI command selection.
- `cli-help-benchmark` verifies safety metadata presence.
- Runtime direction points to `v6.5` without false `v6.4` frontier mismatch warnings.
- AOF records the v6.5 work item boundary through `TASK-070`.
- Safe local reads/writes are preapproved by run contract, while project writes, external writes, and dangerous operations require per-run approval.

## Authority Boundary

The release does not claim:

- tool or platform permission bypass
- automatic approval for `git push`, publish, deploy, secrets, billing, production, destructive, or irreversible operations
- full Mission Control redesign
- complete elimination of context bloat
- semantic proof that every command classification is perfect

The release does claim:

- command category and command safety are now separate surfaces
- safe local `.aof` runtime work can be distinguished from project/external/dangerous work
- `situation-assess` no longer treats the shipped release mentioned in "After v6.4.0, advance v6.5..." as the target frontier
- v6.5 direction is now runtime-backed and visible through current runtime artifacts

## Gates

1. Runtime direction gate: `situation-assess` reports `v6.5` as the active frontier and no truth conflicts.
2. Command safety gate: command registry and command help expose `safety_level` and `approval_policy`.
3. Release surface gate: README, quickstart, package, bootstrap, active release manifest, roadmap, checklist, and notes align to `6.5.0`.
4. Verification gate: command routing, organization, release state, focused runtime tests, full tests, and smoke validation pass.
5. Boundary gate: release docs do not claim automatic approval for external, destructive, publishing, deploy, billing, secrets, production, or irreversible operations.
