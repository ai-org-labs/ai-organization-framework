# AOF v7.3 Release Definition

## Release

- version: `7.3.0`
- tag: `v7.3.0`
- theme: Governed Multi-Actor Pilot
- release task: `TASK-092`

## Release Claim

`v7.3.0` makes AOF multi-actor claims harder to fake by requiring a bounded pilot record before organization work is treated as governed multi-actor execution.

The bounded claim is:

> AOF can record and audit a Governed Multi-Actor Pilot that links a parent orchestrator, Visionary / Builder / Guardian Council presence, actor roster evidence, actor output handoffs, Council judgment, and Work Execution Packet closure.

## Required Outcomes

- `multi-actor-pilot-record` writes schema-valid multi-actor pilot artifacts.
- `multi-actor-pilot-audit` verifies pilot presence, task linkage, accepted status, core Council roles, actor output handoffs, Council judgment, Work Execution Packet ref, resolvable refs, and not-proven boundary.
- `release-state-audit` includes `multi-actor-pilot-audit` for `7.3.0` and later releases.
- TASK-092 has work readiness, context integrity, agent session, actor roster, actor handoff, Council review, Work Execution Packet, Archmap impact, independent evidence, and Multi-Actor Pilot evidence.
- Multi-Actor Pilot does not replace Guardian/Council review; it proves the organization handoff chain is reconstructable.

## Authority Boundary

The release does not claim:

- autonomous workforce performance
- semantic correctness of actor judgment
- external worker adapter readiness
- provider-native child-agent spawning
- parallel lane join/conflict semantics
- external adoption
- Node.js 25+ support

The release does claim:

- multi-actor participation claims can require Council and actor handoff evidence
- Visionary / Builder / Guardian presence can be audited as core Council participation
- actor outputs can be prevented from bypassing organization handoff and Council judgment
- Work Execution Packet can act as the closure unit for a bounded multi-actor pilot

1. Contract gate: pilot and audit schemas exist.
2. Writer gate: `multi-actor-pilot-record` creates schema-valid pilot artifacts.
3. Audit gate: `multi-actor-pilot-audit` fails missing pilots and passes TASK-092 release evidence.
4. Release integration gate: `release-state-audit` includes Multi-Actor Pilot audit for `7.3.0` and later.
5. Governance gate: TASK-092 has work readiness, context integrity, agent session, actor roster, actor output handoffs, Council review, Work Execution Packet, Archmap impact, and independent evidence.
6. Boundary gate: Multi-Actor Pilot is not described as semantic truth, external adoption, autonomous workforce performance, or provider-native child-agent execution.
