# AOF v6.9 Release Definition

## Release

- version: `6.9.0`
- tag: `v6.9.0`
- theme: Executable Pre-Implementation Quality Gates
- parent tasks: `TASK-081`, `TASK-082`, `TASK-083`

## Release Claim

`v6.9.0` makes implementation readiness executable.

The bounded claim is:

> AOF can require a work item to declare goal, risk, loss boundary, acceptance gates, evidence plan, maker/checker separation, and stop conditions before implementation work is treated as ready.

## Required Outcomes

- `work-readiness-record` writes schema-valid readiness records.
- `work-readiness-audit` verifies readiness records for implementation-grade work.
- The audit rejects missing readiness records.
- The audit rejects readiness records without goal, risk, loss boundary, acceptance gates, evidence plan, maker/checker separation, stop conditions, or QIF refs.
- `release-state-audit` includes `work-readiness-audit` for `6.9.0` and later releases.
- `TASK-082` has its own committed readiness record.
- `TASK-083` has release closure readiness evidence.
- The command registry, CLI reference, and tests include the new commands.

## Authority Boundary

The release does not claim:

- semantic truth validation
- automatic QIF verdict computation
- that readiness proves implementation quality
- that readiness replaces Need Validation, Council review, tests, or release-state audit
- governed multi-actor orchestration
- Node.js 25+ support

The release does claim:

- work readiness can be represented as a canonical runtime artifact
- missing readiness gates can fail before release sign-off
- loops must declare stop conditions before execution is treated as ready
- maker/checker separation is inspectable before work begins

## Gates

1. Readiness schema gate: readiness record and audit schemas exist and are covered by tests.
2. Readiness writer gate: `work-readiness-record` creates schema-valid readiness artifacts.
3. Readiness audit gate: `work-readiness-audit` passes committed readiness records and fails missing readiness.
4. Command surface gate: command registry, CLI reference, and AI help include the new readiness commands.
5. Governance gate: `TASK-082` has Council review provenance and independent runtime/test evidence.
6. Release surface gate: package, active release manifest, organization release contract, README, checklist, and notes align to `6.9.0`.
7. Verification gate: work readiness audit, quality ledger audit, release-state audit, organization verification, command routing, CLI help benchmark, decision verification, focused tests, full tests, and smoke validation pass.
8. Boundary gate: readiness pass is not described as semantic truth, implementation quality, or market/product validation.
