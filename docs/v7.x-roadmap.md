# AOF v7.x Roadmap

## Purpose

`v7.x` turns AOF from a framework that can audit final artifacts into a runtime
that can reconstruct how the AI organization actually worked.

The v6 line made governance executable:

- v6.7: governance audits
- v6.8: quality ledger
- v6.9: pre-implementation readiness gates

The v7 line should make live AI work reconstructable and then use that
reconstructable evidence to support safer multi-actor execution.

## North Star

An operator should be able to answer:

- What did the AI organization do?
- Which task, requirement, and goal was it working on?
- Which prompt, response, tool call, artifact write, decision, verification,
  blocker, and stop condition occurred?
- Which actor or council was responsible?
- Which evidence proves the work path happened?
- Which requirements are covered, partially covered, blocked, or not started?
- How much of the requirement-linked task set is done?
- What is the current completion forecast, token/cost forecast, and burn-down
  trend?
- What remains unproven?

without reverse-engineering scattered final artifacts.

Mission Control should eventually answer these questions directly from
canonical runtime artifacts. It must not merely show that tasks exist; it should
show whether the work is converging against declared requirements, quality
intent, budget, stop conditions, and remaining risk.

## Release Sequence

### v7.0.0: Agent Session Observability

Theme:

- Treat agent session events as canonical runtime evidence.

Scope:

- `session-event-record`
- `runtime-event-stream`
- local file-backed event writer
- `session-observability-audit`
- Mission Control projection of latest session stream and audit state
- canonical `mission-control` command that refreshes status and starts the
  viewer from one operator entrypoint

Already completed implementation-grade slices:

- `TASK-085`: agent session event stream
- `TASK-086`: persistent Mission Control session and organization view
- `TASK-087`: Mission Control auto-start operator entrypoint

Remaining release work:

- create v7.0 release definition, checklist, and notes
- update package/bootstrap release refs from v6.9.0 to v7.0.0
- run release-state audit and full verification
- tag and publish release after sign-off

Acceptance gates:

- session streams can link to task, requirement, tests, risks, decisions, and
  release-ready evidence
- session observability audit passes
- Mission Control shows the latest reconstructable session state
- event-stream evidence is not misrepresented as semantic truth

Boundary:

- no provider-specific Copilot dependency
- no live network streaming
- no claim that event reconstruction proves the work was correct

### v7.1.0: Context And Reference Integrity

Theme:

- Prevent runtime work from depending on hidden conversation context or stale
  external references.

Scope:

- context pack integrity audit
- external reference resolution audit
- operational map reference audit
- stale or missing source-of-truth detection
- Mission Control blocker projection for missing context/reference evidence

Why next:

- v7.0 can reconstruct events, but the events are not enough if the actor used
  hidden context or unverifiable external assumptions.

Acceptance gates:

- work items can declare required context and external refs
- audits reject unresolved, stale, or hidden reference dependencies
- Council review can see which references were actually used

Boundary:

- do not attempt full semantic validation of external content
- verify availability, freshness, provenance, and declared use first

### v7.2.0: Work Execution Packet

Theme:

- Make each work item executable as a bounded packet rather than an open-ended
  chat continuation.

Scope:

- work execution packet
- approved run contract
- budget and stop-condition projection
- accepted-change evidence
- discarded-output evidence
- cost-per-accepted-change metric seed
- requirement refs and requirement-derived task refs
- functional / non-functional requirement extraction boundary
- QIF quality intent refs for the work item

Why next:

- v6.9 defines readiness before work starts.
- v7.0 records what happened during work.
- v7.2 should bind the two into a compact execution contract.
- Mission Control cannot forecast progress unless work packets expose which
  requirements they are meant to satisfy and what quality intent governs their
  acceptance.

Acceptance gates:

- every implementation-grade work item can show goal, allowed commands,
  expected outputs, budget/stop conditions, verification plan, and handoff shape
- event stream can be checked against the execution packet
- work packets can link to functional requirement refs, non-functional
  requirement refs, and QIF refs
- missing requirement definition blocks forecast and release-readiness claims

Boundary:

- no automatic optimization loop until manual packet execution is proven
- v7.2 does not need full requirements management; it only needs enough
  requirement linkage to make execution packets forecastable and reviewable

### v7.3.0: Governed Multi-Actor Pilot

Theme:

- Introduce a bounded parent/child actor loop using the v7.0 event stream and
  v7.2 execution packet as evidence.

Scope:

- parent orchestrator assignment packet
- required core council presence: Visionary / Builder / Guardian
- actor roster proposal
- actor work packet
- actor output handoff
- council judgment packet
- loop proof over one small multi-actor work item

Why here:

- multi-actor execution without reconstructable session evidence would scale
  confusion.
- after v7.0-v7.2, actor work can be bounded, observed, and reviewed.

Acceptance gates:

- Council can judge actor roster before execution
- actors submit outputs through organization handoff, not directly as final truth
- parent orchestrator records the loop without hiding child work
- Maker / Checker / Council separation is preserved

Boundary:

- one bounded pilot, not general autonomous workforce
- no claim of true parallelism unless the runtime evidence proves separate lanes

### v7.4.0: Governed Parallel Lanes

Theme:

- Allow multiple actor lanes to progress concurrently while preserving evidence,
  budget, stop, and Council join semantics.

Scope:

- lane IDs
- lane budget and stop conditions
- lane-local event streams
- join record
- conflict and dependency detection
- Mission Control lane view

Acceptance gates:

- parallel lanes cannot merge without join evidence
- missing Guardian or Council review blocks merge
- lane conflicts become visible blockers

Boundary:

- local orchestration first
- do not claim backend autonomous spawning unless implemented by the available
  agent platform

### v7.5.0: Requirements Coverage, Forecasting, And Organization Analytics

Theme:

- Use accumulated readiness, event, review, quality, and outcome evidence to
  explain whether the AI organization is converging on declared requirements.

Scope:

- requirement inventory derived from Need Validation, project charter, QIF
  quality intent, and work execution packets
- functional requirement extraction
- non-functional requirement extraction
- requirement-to-task coverage matrix
- requirement task completion ratio
- blocked / at-risk / unstarted requirement signals
- completion forecast based on accepted-change rate, open requirement tasks,
  blockers, and review churn
- token / cost forecast based on observed session costs or configured budget
  estimates
- burn-down chart data for requirement-linked remaining work
- accepted change rate
- discarded output rate
- cost per accepted change
- review churn
- blocker recurrence
- actor/council workload
- outcome-linked learning signals
- Mission Control summary cards for coverage, forecast, burn-down, and top
  requirement risks

Acceptance gates:

- Mission Control can show requirement coverage, requirement-task completion,
  completion forecast, token/cost forecast, and burn-down from canonical
  artifacts
- requirements distinguish functional and non-functional concerns
- QIF is used to define quality intent, risk, loss boundary, acceptance gates,
  evidence, and verdict boundary for requirements whose quality cannot be
  measured by task completion alone
- metrics are tied to accepted work evidence, not activity volume
- analytics distinguish structural/runtime evidence from semantic or market
  truth
- Mission Control can show top bottlenecks and next improvement candidate
- forecast output states uncertainty and refuses precision when evidence is
  insufficient

Boundary:

- metrics do not prove quality by themselves
- QIF remains the boundary for quality claims
- forecast is an operational estimate, not a release promise
- task completion is not requirement satisfaction unless acceptance evidence and
  QIF boundary are present

### v7.6.0: Provider-Neutral Session Export

Theme:

- Let AOF export session and governance evidence for external tools without
  becoming dependent on one vendor.

Scope:

- JSONL or bundled session export
- optional adapter contract
- redaction and secret boundary
- import/read-only replay surface
- release evidence bundle

Acceptance gates:

- export preserves task, requirement, artifact, decision, test, risk, and stop
  linkages
- export does not leak secrets
- external-write operations remain explicitly approved

Boundary:

- no mandatory GitHub Copilot, Claude, Codex, SIEM, Slack, or Jira integration
- adapters are downstream of local canonical artifacts

### v7.7.0: Adoption-Grade v7 Runtime

Theme:

- Package the v7 runtime so a new project can use session observability and
  bounded work execution without learning the whole framework first.

Scope:

- quickstart rewrite around `mission-control`
- first observed work example
- first multi-actor pilot example if v7.3 is complete
- adoption benchmark extension
- human-readable explanation of AP/NV/QIF/session evidence checks

Acceptance gates:

- a fresh managed project can start Mission Control, run one governed work item,
  and inspect session evidence
- benchmark outputs explain what is checked, expected, and still unproven
- examples are not empty-looking or AOF-self-hosting-only

Boundary:

- adoption proof is still bounded to reproducible first work, not external market
  validation

## v8 Candidate

`v8.0` should only be considered after v7 proves reconstructable, bounded,
multi-actor runtime loops.

Candidate theme:

- Bounded Autonomous Organization Runtime.

Required before v8:

- event streams are reliable
- context/reference integrity is checked
- work execution packets are bounded
- multi-actor pilot is proven
- parallel lanes have join and conflict semantics
- Mission Control can show requirements coverage, requirement-task completion,
  completion forecast, token/cost forecast, burn-down, and bottlenecks from
  canonical artifacts
- QIF-governed requirement quality boundaries are connected to execution and
  forecast evidence
- adoption path is understandable outside AOF self-hosting

## Permanent Boundaries

- Reconstructability is not semantic truth.
- Activity volume is not quality.
- Task completion is not requirement satisfaction without acceptance evidence.
- Forecast precision must not exceed evidence quality.
- Viewer output is not source of truth.
- Mission Control derives from canonical artifacts.
- Requirements, functional requirements, non-functional requirements, quality
  intent, evidence, and verdict boundaries should be explicit before Mission
  Control claims coverage.
- Manual proof comes before automation.
- Loop means `goal + execution + verification + stop condition`.
- Maker, Checker, and Council must remain separated.
- External writes, publishing, deployment, secrets, billing, production mutation,
  and destructive operations require explicit approval.
