# AOF v8.7 Approval Authenticity

Provider execution approval must bind adapter capability, human approval authenticity,
concrete target operation, credential scope, budget, rollback, and stop
conditions before external write can be authorized.

## Problem Fixed From v8.6

`v8.6.0` introduced the Provider Execution Approval Bridge, but the self-hosting
preflight record overclaimed readiness:

- it marked `external_write_authorized: true` while the referenced adapter was
  read-only
- it used Council review evidence as if it were independent human approval
- it did not bind approval to a concrete provider/resource/operation/payload
  target
- credential scope, budget, and rollback boundaries were not structured enough
  for audit

The v8.7 correction is to make external write authorization fail unless all of
those boundaries are explicit and cross-checked.

## Release Contract

External write authorization requires all of the following:

- the referenced adapter explicitly supports `external_write`
- requested `allowed_operations` are a subset of the adapter `operation_modes`
- `human_approval_ref` resolves to a `human-approval-record`
- the human approval is active, approved, and recorded as human approval rather
  than Council self-review
- `target_operation_ref` resolves to a `provider-operation-target-record`
- the human approval target matches the provider operation target
- the approved scope hash matches the target payload hash
- `credential_scope` is non-empty for write-class execution
- `budget` is structured and bounded
- `rollback` is structured and supported

## Corrected v8.6 Record

The historical v8.6 preflight artifact is retained on main as a corrected
blocked record:

- artifact: `.aof/artifacts/provider-execution-approvals/PEA-TASK-106-V86-PREFLIGHT.json`
- corrected decision: `blocked`
- corrected execution mode: `read_only`
- corrected external write state: `external_write_authorized: false`

This preserves traceability without letting an invalid approval bridge remain
green.

## v8.7 Proof Boundary

The v8.7 self-hosting proof is still preflight governance evidence. It does not
create a live GitHub issue, does not prove provider output correctness, and does
not prove production execution safety.

It proves a narrower claim: AOF can now distinguish a structurally valid
external-write preflight from a read-only adapter overclaim before production
execution is attempted.
