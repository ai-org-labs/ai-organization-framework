# AOF v6.8 Direction Review

## Runtime Basis

- date: `2026-07-03`
- source task: `TASK-077`
- active release: `6.7.0`
- runtime stage before selection: `frontier-definition-needed`
- release gate: `release-state-audit` passed for `6.7.0`
- governance gates: Archmap impact, review provenance, and evidence independence audits passed for `TASK-071` onward

## Question

What should AOF do after `v6.7.0`?

The candidate from the bounded completion roadmap is:

> `v6.8` — Executable Quality Ledger

## Diagnosis

`v6.7.0` made governance claims harder to fake by requiring machine-checkable evidence for architecture impact, Council review provenance, and evidence independence.

The remaining gap is not another review artifact. The gap is that corrected assumptions, bugs, contradicted claims, semantic uncertainty, and operator feedback are still too easy to record as prose but too hard to audit as a living ledger.

Without an executable ledger, AOF can still regress into:

- treating a one-time QIF document as current truth
- losing bug/correction history across work-item boundaries
- claiming review happened without citing a recent runtime run
- failing to escalate low-confidence or contradicted claims
- forcing humans to reconstruct why a verdict changed

## Decision

Select `v6.8` as **Executable Quality Ledger**.

The bounded release claim should be:

> AOF can append quality-relevant evidence events to a canonical ledger, audit whether review/release claims cite recent runtime evidence, and make corrected assumptions or contradicted claims visible before the next governed decision.

## First Implementation Slice

Open the next implementation task as:

`TASK-078`: `v6.8 frontier: executable quality ledger foundation`

Scope:

- add a canonical quality-ledger event schema
- add a `qif-evidence-append` or equivalent runtime writer
- add a `qif-ledger-audit` safe_read verifier
- add fixtures proving missing runtime evidence is detectable
- keep verdict computation out of scope

## Explicit Non-Goals

- Do not auto-compute QIF verdicts.
- Do not claim semantic truth from ledger presence.
- Do not vendor QIF as an internal fixed copy.
- Do not open multi-actor orchestration work; that remains a later v7.0 concern.
- Do not claim operator validation until real operator evidence exists.

## Acceptance Gate

`v6.8` should be considered viable only if:

- ledger events are append-only or at least tamper-visible
- events link to Quality Intent, evidence refs, task refs, and runtime refs
- an audit can fail a review/release claim with no recent runtime evidence
- corrected assumptions and contradicted claims can be represented explicitly
- the ledger remains evidence, not truth

## Council Recommendation

Proceed with `v6.8` as the next frontier.

The first implementation task should build the smallest executable ledger foundation and one negative audit path. This is higher leverage than starting `v6.9` adoption or `v7.0` multi-actor orchestration because both depend on trustworthy quality/provenance history.
