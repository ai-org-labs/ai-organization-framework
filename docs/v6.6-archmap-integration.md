# AOF v6.6 Archmap Integration

## Purpose

AOF should build and maintain an architecture map while development work moves forward.

This is not only a viewer feature. The rule is:

> If AOF changes the organization, runtime, command surface, artifact contract, UI, or external integration, the architecture map must either be updated or explicitly marked as unaffected.

Archmap is the diagram language and renderer for that map. `@archmap/icons` is the optional icon source for provider and service-specific architecture nodes.

References:

- Archmap: `https://github.com/ai-org-labs/archmap/tree/v0.1.1`
- Archmap icons: `https://github.com/ai-org-labs/archmap-icons/tree/v0.1.2`

## Provider Baseline

AOF tracks Archmap `v0.1.1` as the current external provider baseline.

This adds upstream ScreenFlow / Prototype view support, shared diagram tags controls, curated playground samples, export coverage, and stream input APIs. AOF treats those as available external capabilities, not as automatic AOF runtime claims.

The AOF boundary remains unchanged:

- Archmap source files and AOF artifacts are the source of truth.
- Mission Control may project architecture state, but it does not own truth.
- ScreenFlow / Prototype views should be adopted only when a governed AOF work item needs them.
- Icon availability improves recognition only; it is not quality evidence.

## Product Requirement

AOF must make architectural change visible as work happens.

For each governed work item, the runtime should be able to answer:

- What architecture component changed?
- What new component, dependency, boundary, actor, data flow, command, or external reference was introduced?
- What existing component became riskier or more important?
- What map source proves the current architecture view?
- Did Council approve the map impact, or was the map marked unaffected?

## Runtime Flow

The Archmap-aware AOF flow is:

```text
Need / Work Item
-> Need Validation or Work Governance
-> Architecture Impact Check
-> Archmap Source Update or Unaffected Decision
-> Council-ready Handoff
-> Council Review
-> Approved Work
-> Mission Control / Human Recognition projection
```

## Mandatory Contract

Every implementation-grade work item should carry one of these outcomes:

1. `archmap_update_required`
2. `archmap_unaffected`
3. `archmap_deferred_with_reason`

`archmap_update_required` must include:

- Archmap source ref
- changed node / edge / zone / boundary refs
- reason for change
- owner actor
- Council review status

`archmap_unaffected` must include:

- reason why architecture did not change
- reviewer role
- evidence ref

`archmap_deferred_with_reason` must include:

- missing information
- risk of deferral
- follow-up task or blocker

## Archmap Artifact Locations

Recommended AOF-managed locations:

- `docs/archmaps/*.archmap`: human-readable Archmap source
- `.aof/artifacts/archmap/impact/*.json`: work-item-level architecture impact records
- `.aof/artifacts/archmap/decisions/*.json`: Council map approval or deferral decisions
- `.aof/artifacts/visibility/current/mission-control.json`: projection of current map state, not source of truth

## Archmap Source Rules

Archmap source should remain small enough to review.

- Use the graph section for visible topology.
- Use YAML metadata for semantic facts.
- Use `zone` for ownership or runtime area.
- Use `boundary` for trust, permission, approval, or external integration boundaries.
- Use `provider` and `kind` only when an icon meaning is intentional.
- Do not use icons as proof of integration.
- Do not render decorative nodes without canonical AOF artifact evidence.

## Icon Rules

`@archmap/icons` is optional.

Icons may improve recognition, but they are not quality evidence by themselves.

Use icons when:

- the node represents a known provider, product, cloud service, or platform
- the provider/kind mapping is known
- the diagram remains understandable without the icon pack

Do not block AOF runtime work because an icon is unavailable.

## Mission Control Boundary

Mission Control may render Archmap views, but it must not become the source of truth.

The source of truth remains:

- Archmap source files
- AOF work item artifacts
- Council review artifacts
- verification artifacts

Mission Control should show:

- current architecture map ref
- last map-changing work item
- pending map impact decisions
- map blockers
- whether current work is architecture-affecting

## v6.6 Acceptance Gate

v6.6 should be considered complete only if:

- AOF has a documented Archmap integration contract.
- A committed Archmap source exists for the current AOF runtime architecture.
- A current work item can declare `archmap_update_required`, `archmap_unaffected`, or `archmap_deferred_with_reason`.
- Mission Control can surface the current map ref and map-impact state without owning truth.
- Verification can detect a missing map-impact decision for implementation-grade work.

## Non-Goals

v6.6 does not need to:

- fully embed Archmap rendering into Mission Control
- require `@archmap/core` as a runtime dependency
- require `@archmap/icons` for all diagrams
- generate perfect architecture diagrams automatically
- prove architecture quality from diagram existence alone

The first release-grade value is governed architectural traceability, not visual polish.
