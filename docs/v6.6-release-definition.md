# AOF v6.6 Release Definition

## Release

- version: `6.6.0`
- tag: `v6.6.0`
- theme: Archmap-Aware Mission Control
- parent task: `TASK-071`

## Release Claim

`v6.6.0` makes architecture impact a governed part of AOF work.

The bounded claim is:

> AOF can require implementation-grade work items to update, defer, or explicitly mark architecture impact as unaffected; keep Archmap source as the source of truth; and project the current architecture map and impact state into Mission Control without letting the viewer become proof.

## Required Outcomes

- Archmap integration is documented in `docs/v6.6-archmap-integration.md`.
- A committed Archmap source exists at `docs/archmaps/aof-runtime-current.archmap`.
- The current implementation-grade work item records an Archmap impact decision in `.aof/artifacts/archmap/impact/TASK-071.json`.
- Mission Control projects the current map ref and map-impact state from canonical artifacts.
- QIF provider alignment remains on QIF `v0.3.1` while machine-readable package compatibility remains explicit.
- Node.js 22 and Node.js 24 are CI-validated runtime lanes.
- `TASK-071` is closed with Council disposition for the Archmap impact.

## Authority Boundary

The release does not claim:

- full Archmap renderer integration into Mission Control
- `@archmap/core` or `@archmap/icons` as required runtime dependencies
- visual architecture comprehension is externally validated
- architecture quality is proven because a diagram exists
- QIF `v0.3.1` provides executable verifier replacement artifacts
- Node.js 25+ support

The release does claim:

- architecture impact is now part of governed work-item closure
- Archmap source and AOF artifacts remain canonical evidence
- Mission Control is a read-only projection of map state
- Council disposition is required before Archmap-impact work is released
- Node.js 24 is officially covered by CI alongside Node.js 22

- CI workflow: `.github/workflows/ci.yml`

## Gates

1. Runtime direction gate: `situation-assess` reported `TASK-071` as the v6.6 frontier before closure and no truth conflicts.
2. Archmap contract gate: implementation-grade work can declare `archmap_update_required`, `archmap_unaffected`, or `archmap_deferred_with_reason`.
3. Council disposition gate: `TASK-071` Archmap impact is approved, rejected, or deferred before release.
4. Mission Control boundary gate: Mission Control projects map state from canonical artifacts and does not own truth.
5. Runtime support gate: Node.js 22 and Node.js 24 are CI lanes; Node.js 25+ remains outside the supported lane.
6. Release surface gate: README, quickstart, package, bootstrap, active release manifest, roadmap, checklist, and notes align to `6.6.0`.
7. Verification gate: organization, release state, command routing, decision verification, focused runtime tests, full tests, and smoke validation pass.
