# v8.1 Direction Review

## Runtime Basis

- task: `TASK-101`
- session: `SESS-V81-DIRECTION`
- previous release: `v8.0.0`
- selected frontier: `v8.1 Provider Adapter Governance`

## Decision

Proceed with a bounded v8.1 release that adds provider adapter governance as a
contract and audit surface.

## Why This Is The Right Slice

`v8.0.0` made external actors, tools, providers, and references governable as
runtime resources. The next unsafe gap is not provider execution itself; it is
the adapter layer that can silently turn an external provider into authority.

If AOF adds provider execution before adapter boundaries, it risks treating
provider availability as permission, freshness, truth, and safe side effects.
v8.1 therefore makes provider adapters auditable before automation grows.

## Selected Scope

Included:

- `provider-adapter-record`
- `provider-adapter-audit`
- schema validation for provider adapter records and audits
- release-state gate for `8.1.0` and later
- Mission Control projection from provider adapter audit artifacts
- self-hosting adapter evidence for this repository

Deferred:

- autonomous external provider execution
- credential handling
- billing or deployment adapters
- hosted provider orchestration
- semantic correctness of provider outputs

## Council Judgment

- Visionary: proceed because adapter governance is the next leverage point after
  external runtime resources.
- Builder: proceed with a narrow record/audit implementation rather than a broad
  execution engine.
- Guardian: proceed only if write-capable adapters fail without escalation and
  if adapter readiness is not treated as semantic truth.

## Release Boundary

v8.1 can claim structural/runtime provider adapter governance. It cannot claim
that external providers are safe, correct, available, economical, or approved
for production use.
