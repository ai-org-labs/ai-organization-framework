# AOF v6.9 Direction Review

## Runtime Basis

- date: `2026-07-06`
- source task: `TASK-081`
- active release: `6.8.0`
- runtime stage before selection: `frontier-definition-needed`
- release gate: `release-state-audit` passed for `6.8.0`
- recent runtime evidence: `TASK-080` updated the Archmap provider baseline and kept the post-v6.8 frontier open

## Question

What should AOF do after `v6.8.0`?

The runtime-visible candidates are:

1. executable pre-implementation quality gates
2. standalone context / reference audits
3. first bounded step toward governed multi-actor orchestration

## Diagnosis

`v6.8.0` made quality-relevant evidence appendable and auditable, but it still mostly catches problems after a work item has already produced output.

The highest-leverage remaining gap is earlier:

- work can begin before the success expectation is explicit
- a task can be “done” without a predeclared loss boundary
- evidence can be gathered after the fact instead of planned before implementation
- maker/checker separation can be asserted but not required before execution
- loops can run without explicit stop conditions

This is why user-facing validation still feels opaque: AOF can say that audits passed, but it does not always expose what the work was expected to prove before implementation began.

## Decision

Select `v6.9` as **Executable Pre-Implementation Quality Gates**.

The bounded release claim should be:

> AOF can require a work item to declare goal, risk, loss boundary, acceptance gates, evidence plan, maker/checker separation, and stop conditions before implementation work is treated as ready.

## Why Not The Other Candidates First

Standalone context / reference audits are important, but they become more useful once work readiness already defines which context and references matter.

Governed multi-actor orchestration is still the larger v7-class frontier. Starting it before executable readiness gates would scale unclear work faster instead of making work better.

## First Implementation Slice

Open the next implementation task as:

`TASK-082`: `v6.9 frontier: executable work readiness gate`

Scope:

- add a canonical work-readiness record schema
- add `work-readiness-record`
- add `work-readiness-audit`
- prove the audit fails missing goal, loss boundary, acceptance gates, evidence plan, maker/checker separation, or stop conditions
- record the readiness gate for `TASK-082` itself

## Explicit Non-Goals

- Do not implement governed parallel actors.
- Do not compute QIF verdicts automatically.
- Do not prove semantic truth from readiness records.
- Do not replace Need Validation.
- Do not make readiness pass a substitute for Council review or release-state audit.

## Acceptance Gate

`v6.9` should be considered viable only if:

- readiness records are schema-valid runtime artifacts
- audits can fail a work item before implementation readiness is claimed
- records link to task refs, QIF refs, evidence plans, and stop conditions
- maker/checker separation is explicit
- release-state audit continues to pass with the new gate evidence

## Council Recommendation

Proceed with `v6.9` as the next frontier.

The first implementation task should build the smallest executable readiness gate and one negative audit path. This is the correct bridge between the quality ledger and future multi-actor orchestration.
