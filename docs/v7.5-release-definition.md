# AOF v7.5 Release Definition

## Release

- version: `7.5.0`
- tag: `v7.5.0`
- theme: Requirements Coverage, Forecasting, And Organization Analytics
- release task: `TASK-094`

## Release Claim

`v7.5.0` makes AOF progress and forecast claims auditable by requiring requirement coverage records before release readiness can be claimed.

The bounded claim is:

> AOF can record and audit a Requirement Coverage Record that links functional, non-functional, QIF quality-intent, and release-gate requirements to work items, source refs, evidence refs, coverage counts, blocker state, forecast boundaries, and explicit not-proven limits.

## Required Outcomes

- `requirement-coverage-record` writes schema-valid requirement coverage artifacts.
- `requirement-coverage-audit` verifies coverage record presence, task linkage, accepted status, requirement entries, source refs, linked work refs, evidence refs, consistent coverage counts, forecast boundary, optional burndown ref, and not-proven boundary.
- `release-state-audit` includes `requirement-coverage-audit` for `7.5.0` and later releases.
- TASK-094 has work readiness, context integrity, agent session, actor roster, actor output handoff, Council review, Work Execution Packet, Multi-Actor Pilot, Parallel Lane Pilot, Archmap impact, independent evidence, and Requirement Coverage evidence.
- Requirement coverage and forecast data are treated as bounded planning evidence, not as proof of semantic requirement satisfaction or delivery certainty.

## Authority Boundary

The release does not claim:

- semantic satisfaction of requirements
- forecast accuracy
- market truth
- external adoption
- provider-neutral session export
- autonomous scheduling
- Node.js 25+ support

The release does claim:

- progress claims can be tied to explicit requirements
- requirement coverage can be linked to work items and evidence
- task completion and requirement coverage are separate concepts
- blockers and forecast boundaries can be made auditable
- release readiness can reject missing requirement coverage evidence

## Gates

1. Contract gate: record and audit schemas exist.
2. Writer gate: `requirement-coverage-record` creates schema-valid coverage artifacts.
3. Audit gate: `requirement-coverage-audit` fails missing coverage and passes TASK-094 release evidence.
4. Release integration gate: `release-state-audit` includes Requirement Coverage audit for `7.5.0` and later.
5. Governance gate: TASK-094 has work readiness, context integrity, agent session, actor roster, actor output handoffs, Council review, Work Execution Packet, Multi-Actor Pilot, Parallel Lane Pilot, Archmap impact, and independent evidence.
6. Boundary gate: coverage and forecast are not described as semantic truth, external validation, market truth, delivery certainty, or adoption proof.
7. Verification gate: focused runtime tests, governance audits, release-state audit, organization verification, command routing, CLI help benchmark, npm test, and smoke validation pass.
