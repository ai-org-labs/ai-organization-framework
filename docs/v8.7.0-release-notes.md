# AOF v8.7.0 Release Notes

## Summary

`v8.7.0` closes the approval authenticity gap found after `v8.6.0`.

The key change is that AOF no longer treats an approval-shaped artifact as enough
to authorize external write. The approval must align with adapter capability,
human approval evidence, a concrete target operation, credential scope, budget,
rollback, and stop-condition boundaries.

## Added

- `provider-operation-target-record`
- `human-approval-record`
- structured provider execution approval fields:
  - `target_operation_ref`
  - `credential_scope`
  - `budget`
  - `rollback`
- audit checks for:
  - adapter `operation_modes` compatibility
  - active human approval authenticity
  - target operation resolution
  - approved payload scope alignment
  - write-class credential scope
  - structured budget boundary
  - rollback support

## Fixed

- Corrected the v8.6 self-hosting preflight record from external-write approval
  to blocked/read-only, because the referenced adapter was read-only and the
  approval evidence was Council review rather than independent human approval.
- Added regression coverage so read-only adapters cannot pass external-write
  approval audit.

## Not Proven

`v8.7.0` does not prove live external execution.

Not proven:

- live GitHub issue creation
- provider output correctness
- credential safety beyond declared scope
- billing safety beyond declared budget
- rollback execution success
- autonomous production execution

## Operator Reading

Human-facing interpretation:

> v8.7 makes AOF stricter about what counts as permission to write outside the
> repository. It is no longer enough to say "approved"; AOF now checks whether
> the adapter can write, whether a human approved the exact target, and whether
> credential, budget, rollback, and stop boundaries are recorded.
