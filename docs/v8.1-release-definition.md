# AOF v8.1 Release Definition

## Release Theme

External Reference And Provider Adapter Governance.

Human-facing wording:

> AOF can now show when a provider adapter is bounded enough to use, and can
> reject adapter paths that hide write authority, stale context, missing approval,
> or side effects.

## Required Outcomes

- `provider-adapter-record` writes a governed adapter contract.
- `provider-adapter-audit` checks adapter presence, refs, boundaries, freshness,
  approval policy, escalation, provenance, and not-proven limits.
- Write-capable adapter modes fail without explicit escalation.
- Dangerous adapter modes cannot be marked `ready`.
- `release-state-audit` includes `provider-adapter-audit` for `8.1.0` and later.
- Mission Control includes provider adapter projection from canonical audit
  artifacts.
- v8.0 external resource/use governance remains compatible.

## Authority Boundary

Provider adapters are access-path contracts. They do not prove provider output
truth, credential safety, billing safety, production safety, or operator
acceptance.

## Acceptance Gates

1. Provider adapter command help is AI-readable.
2. Provider adapter records and audits validate against bundled schemas.
3. Focused provider adapter tests pass for pass and fail cases.
4. Self-hosting provider adapter audit passes.
5. `release-state-audit` includes and passes the provider adapter gate.
6. `organization-verify`, `command-routing-audit`, `cli-help-benchmark`,
   `npm test`, and `npm run smoke` pass.
