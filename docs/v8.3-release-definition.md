# AOF v8.3 Release Definition

## Theme

Mission Control Operator Acceptance Evidence.

`v8.3.0` makes operator validation visible as release-grade Mission Control
evidence. The operator should be able to see acceptance, rejection,
clarification need, reproduction state, governance action, and evidence refs
without opening raw JSON first.

## Required Outcomes

- Mission Control projects operator validation status from
  `operator-validation-audit`.
- Operator acceptance states are visually distinct: accepted, rejected,
  needs clarification, not reproduced, missing, and invalid.
- Operator validation records include feedback summary, Mission Control ref,
  work item ref, evidence refs, governance action, blocking reason, and
  not-proven boundary.
- The viewer exposes an Operator Acceptance Evidence panel with clickable audit
  and record details.
- Release-state remains compatible with v8.2 operator validation, v8.1 provider
  adapter governance, and v8.0 external resource governance.

## Non-Goals

- This release does not prove market adoption.
- This release does not prove semantic correctness.
- This release does not authorize autonomous external writes.
- This release does not add production credential, billing, or deployment
  automation.

## Acceptance Boundary

AOF may claim operator acceptance visibility only when the Mission Control
projection is derived from canonical operator validation artifacts and the audit
record includes the evidence path and not-proven boundary.

If operator feedback is rejected, unclear, not reproduced, or missing, Mission
Control must expose that state rather than treating the release as accepted.
