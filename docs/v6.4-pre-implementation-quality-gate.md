# AOF v6.4 Pre-Implementation Quality Gate

## Source Feedback

This direction is based on an external project retrospective covering `v0.52.0` through `v0.56.0`.

The failure was not that AOF/QIF was unused. The failure was that AOF/QIF was used too much as post-implementation evidence and structural validation, and not enough as a pre-implementation gate that challenges product assumptions before code is written.

## Core Diagnosis

AOF/QIF already had concepts that could have caught the issue earlier:

- QIF Loss Boundary: the canvas becomes the hidden source of truth
- QIF MVP honesty
- QIF UI verification
- AOF operating rule: changes connect to state, canonical artifacts, QIF evidence, and Git baseline
- Version gate: changelog, AOF state, QIF evidence, verification, commit, and tag align

The missing behavior was operational:

> AOF did not force the orchestrator to ask the right pre-implementation questions before accepting a solution-shaped task.

A second retrospective exposed the same pattern from a different angle:

> AOF/QIF worked as records, but not as done-before forcing functions.

The failure mode was not uniform carelessness. It was asymmetric rigor:

- strict inside the focused implementation surface
- weak on adjacent surfaces
- weak on downstream/public replay paths
- weak on user-visible E2E observation
- weak on rubric-based visual or comprehension scoring

## Product Failure Pattern

The external project first implemented a blank/sample canvas switch. Later, it corrected the model to project-file based startup:

- new project: fully empty project data
- sample project: explicit sample URL or project file
- seed model: canonical validation fixture
- editor: work surface
- analysis projection: read-only view

The earlier implementation passed local checks, but the core assumption was wrong. It solved a data-boundary problem with a UI toggle.

## v6.4 Release Direction

v6.4 should expand the current QIF-Governed Benchmark Explanation candidate into:

> Pre-Implementation Quality Gate and QIF-Governed Explanation.

This means AOF should not only explain benchmark results after execution. It should also require a short quality gate before implementation when a change affects user-facing UI, data boundaries, sample/new/template/seed semantics, or product assumptions.

## Required Gate

Before implementation, a work item that touches product assumptions must record:

1. User problem
2. Product assumption
3. Data model impact
4. UI responsibility
5. Sample / new project / template / seed impact
6. Alternative approaches
7. Negative acceptance criteria
8. QIF evidence plan
9. Browser or operational verification plan
10. Stop / defer / experiment condition

Before declaring done, the work item must also record:

1. User-visible E2E evidence
2. Adjacent surface impact review
3. Downstream/public path review
4. Evidence independence grade
5. Guardian fixed-question answers
6. QIF ledger update or explicit no-change rationale

## Negative Acceptance

Positive acceptance is not enough.

For UI/product work, QIF acceptance must include states that must not be allowed. Examples:

- sample data is not shown as the default new project state
- project loading responsibility is not replaced by a UI toggle
- editor does not contain read-only analysis panels
- concept density does not make first-use comprehension impossible
- sample, seed, template, new project, and user project are not conflated

## AOF Runtime Additions

v6.4 should introduce or formalize:

- `problem-before-solution` gate
- `product-assumption-review`
- `negative-acceptance` requirement
- `corrected-assumption` artifact or field
- `first-use-scenario` gate
- evidence independence grading
- done-before Guardian fixed-question review
- rubric-based visual/render/comprehension review when UI output is involved
- QIF ledger update on bug, correction, or rejected assumption
- QIF-governed benchmark explanation output

## Evidence Grading

AOF must not treat all evidence as equal.

For done/release claims, evidence should be graded by independence:

- low independence: code reading, symmetry reasoning, local assumption, or "same function should handle it"
- medium independence: unit or integration test that exercises the touched path
- high independence: user-visible E2E observation of the actual surface/path being claimed
- highest independence: reproduction by a separate actor, browser/user observation, or operational feedback

Code reading and symmetry reasoning are useful, but they must be downgraded when the claim is about what a user sees or what a public URL/path does.

## Done-Before Guardian Questions

Guardian review should not only validate the focused change.

Before done, Guardian must answer:

1. What other path could break even if the touched function works?
2. What adjacent surface depends on the same assumption?
3. Does this create public, irreversible, published, or user-visible risk?

If any answer is uncertain, the work item must either add evidence, defer the claim, or escalate to Council.

## Rubric-Based Visual Review

When output is visual, browser-based, or user-comprehension-sensitive, AOF must not accept a screenshot merely because it exists.

The review must score or explicitly judge:

- grounding: is the visible surface the actual claimed surface?
- visibility: are the important elements visible without hidden scroll or crop assumptions?
- size/density: can the intended user read and use it?
- transparency/occlusion: are overlays, modals, or backgrounds hiding important state?
- composition/responsibility: is this a work surface, projection, governance view, or sample view?

The result can be lightweight, but it must be explicit.

## Living QIF Ledger

QIF should not be a one-time post-release document.

When a bug, user correction, or contradicted assumption appears, AOF should:

1. identify the related Quality Intent
2. update or create evidence
3. revise confidence or uncertainty
4. add negative acceptance if the bug represents a forbidden state
5. create or update follow-up task if residual risk remains

If no QIF update is needed, the work item should state why.

## Trigger Conditions

The gate is required when a work item touches any of:

- sample data
- new project behavior
- templates
- seed fixtures
- project loading
- editor surface
- projection view
- governance view
- user-facing first-use flow
- terminology visible to first-time users

## Corrected Assumption Loop

When a later user comment overturns a previous completed task, AOF should not silently overwrite history.

It should record:

- original assumption
- where it appeared
- why it passed earlier gates
- what later evidence contradicted it
- corrected assumption
- affected tasks/releases
- new negative acceptance criterion

This turns correction into learning rather than blame or unstructured churn.

## QIF Boundary

QIF validator pass remains structural evidence only.

For v6.4, the quality claim should be:

> AOF can make pre-implementation quality assumptions explicit, attach negative acceptance criteria, explain benchmark meaning through QIF, and escalate semantic uncertainty instead of hiding it.

It should not claim:

- semantic truth
- market truth
- UI comprehension without human/browser evidence
- product correctness merely because tests or artifacts pass

## Release Success

v6.4 is successful if a future solution-shaped request can be stopped or reframed before implementation when:

- the user problem is unclear
- the task title contains the proposed solution rather than the problem
- sample/new/template/seed boundaries are affected but not specified
- negative acceptance is missing
- QIF evidence plan is missing
- browser verification only checks DOM existence and not comprehension or responsibility boundaries
