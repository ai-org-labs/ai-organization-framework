# AOF v7.8 Release Definition

## Release

- version: `7.8.0`
- tag: `v7.8.0`
- theme: Mission Control Coverage Forecast Projection
- release task: `TASK-097`

## Release Claim

`v7.8.0` turns Mission Control from a status viewer into a read-only projection of release-critical governance evidence.

The bounded claim is:

> AOF Mission Control can show requirement coverage, remaining-work forecast, adoption proof status, evidence completeness, and blocking source gaps from canonical runtime artifacts, and release readiness can fail when those projection sources are missing or stale.

## Required Outcomes

- Mission Control includes `requirement_coverage_projection`.
- Mission Control includes `adoption_proof_projection`.
- Mission Control includes `evidence_completeness_projection`.
- The projection identifies the latest requirement coverage record and forecast boundary.
- The projection identifies adoption proof benchmark status and evidence refs.
- The projection identifies required source classes: requirement coverage, adoption proof, agent session, context integrity, release definition, and Archmap impact.
- Missing projection sources are represented as explicit blocking state.
- A dedicated `mission-control-projection-audit` command verifies the projection contract.
- `release-state-audit` includes `mission-control-projection-audit` for `7.8.0` and later releases.
- TASK-097 has runtime evidence tying the release claim to implementation, governance review, requirement coverage, session export, and Mission Control projection audit evidence.

## Authority Boundary

The release does not claim:

- Mission Control visual design is final
- forecast estimates are delivery guarantees
- coverage counts prove semantic truth
- adoption proof proves real market adoption
- provider-native streaming is captured directly
- Node.js 25+ support

The release does claim:

- Mission Control is a read-only projection of canonical AOF artifacts
- requirement coverage and forecast can be surfaced without reading raw JSON manually
- missing evidence can be shown as release-blocking state
- release-state audit can require Mission Control projection evidence

## Gates

1. Projection contract gate: Mission Control schema includes coverage, adoption, and evidence completeness projections.
2. Runtime gate: `mission-control-projection-audit` verifies required projection sources and source refs.
3. Release integration gate: `release-state-audit` includes Mission Control projection audit for `7.8.0` and later.
4. Governance gate: TASK-097 has work readiness, context integrity, agent session, Council review, Work Execution Packet, Multi-Actor Pilot, Parallel Lane Pilot, Requirement Coverage, Session Export, and Archmap disposition evidence.
5. Boundary gate: projection is documented as read-only evidence projection, not semantic truth or forecast certainty.
6. Verification gate: focused runtime tests, governance audits, release-state audit, organization verification, command routing, CLI help benchmark, npm test, and smoke validation pass.
