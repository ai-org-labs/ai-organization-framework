# AOF v7.4 Release Definition

## Release

- version: `7.4.0`
- tag: `v7.4.0`
- theme: Governed Parallel Lanes
- release task: `TASK-093`

## Release Claim

`v7.4.0` makes AOF parallel-work claims auditable by requiring a bounded lane pilot before parallel execution is treated as governed.

The bounded claim is:

> AOF can record and audit a Governed Parallel Lane Pilot that links multiple bounded lanes, lane-local input/output/verification evidence, join/conflict semantics, Council merge/stop/defer/reopen authority, Work Execution Packet closure, and upstream Multi-Actor Pilot evidence.

## Required Outcomes

- `parallel-lane-record` writes schema-valid parallel lane pilot artifacts.
- `parallel-lane-audit` verifies pilot presence, task linkage, accepted status, at least two lanes, lane-local evidence refs, join decision, joined lane coverage, Council decision, Work Execution Packet ref, upstream Multi-Actor Pilot ref, and not-proven boundary.
- `release-state-audit` includes `parallel-lane-audit` for `7.4.0` and later releases.
- TASK-093 has work readiness, context integrity, agent session, actor roster, actor output handoff, Council review, Work Execution Packet, Multi-Actor Pilot, Archmap impact, independent evidence, and Parallel Lane Pilot evidence.
- Parallel Lane Pilot does not replace Council judgment; it records what the lanes did and how Council joined, stopped, deferred, or reopened the work.

## Authority Boundary

The release does not claim:

- autonomous scheduling
- speed improvement
- semantic correctness of lane outputs
- external worker adapter readiness
- provider-native parallel child-agent execution
- external adoption
- Node.js 25+ support

The release does claim:

- parallel-work claims can require lane-local evidence
- lane completion is not the same as parent work completion
- parent work can be joined only through explicit join/conflict semantics
- Council merge/stop/defer/reopen authority can be made auditable
- Work Execution Packet and Multi-Actor Pilot evidence remain upstream gates

## Gates

1. Contract gate: pilot and audit schemas exist.
2. Writer gate: `parallel-lane-record` creates schema-valid pilot artifacts.
3. Audit gate: `parallel-lane-audit` fails missing pilots and passes TASK-093 release evidence.
4. Release integration gate: `release-state-audit` includes Parallel Lane audit for `7.4.0` and later.
5. Governance gate: TASK-093 has work readiness, context integrity, agent session, actor roster, actor output handoffs, Council review, Work Execution Packet, Multi-Actor Pilot, Archmap impact, and independent evidence.
6. Boundary gate: Parallel Lane Pilot is not described as semantic truth, autonomous scheduling, speed improvement, provider-native child-agent execution, or external adoption.
7. Verification gate: focused runtime tests, governance audits, release-state audit, organization verification, command routing, CLI help benchmark, npm test, and smoke validation pass.
